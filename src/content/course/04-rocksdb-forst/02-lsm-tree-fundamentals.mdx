# LSM-tree fundamentals — основа RocksDB

RocksDB — это implementation **Log-Structured Merge tree** (LSM-tree). Без понимания LSM tuning RocksDB — гадание на кофейной гуще. Этот урок препарирует LSM-tree до уровня, на котором можно osмысленно интерпретировать metrics и принимать tuning решения.

LSM — фундаментально разный от B-tree (Postgres, MySQL InnoDB). B-tree оптимизирован для random reads/updates, LSM — для high write throughput, sequential disk I/O.

### MemTable как skiplist

MemTable по умолчанию — **skiplist**: probabilistic balanced data structure с O(log n) insert/read, lock-free для concurrent reads. Альтернативы: hash-skip-list, vector — но skiplist дефолтный потому что balance.

```
write_buffer_size:           64MB     # размер MemTable до flush
max_write_buffer_number:     2        # max immutable MemTable
min_write_buffer_to_merge:   1        # минимум для flush
```

Параметры существенно влияют на write throughput:
- Маленький `write_buffer_size` (8MB) → частые flush → много мелких L0 SST → много compactions
- Большой (256MB) → больше memory usage, больше потерь при crash без WAL (Flink: OK, потому что checkpoint)

### Almost free writes

LSM делает write — это append в MemTable (O(log n) в skiplist) + optional WAL append. Никаких random disk I/O, никаких index updates. Это **why LSM делает 5-10x faster writes** чем B-tree.

## SSTable: immutable, sorted, on-disk

Когда MemTable flushes, Flink RocksDB записывает **SSTable** (Sorted String Table):

```
SSTable file format:
[Data Block 1] [Data Block 2] ... [Data Block N]
[Index Block]   ← maps key range → data block offset
[Bloom Filter Block]   ← per-block bloom filters
[Metadata]      ← min/max keys, sequence numbers, etc.
[Footer]
```

Key properties:
- **Immutable** — once written, never modified. Это критично для checkpoint sharing!
- **Sorted** — entries в порядке возрастания key
- **Block-compressed** — каждый data block (default 4KB) сжат Snappy/ZSTD
- **Bloom filter** — probabilistic «is key here» check (1-byte per key, 1% false positive)
- **Index block** — позволяет O(1) lookup data block для key range

### Block Cache — share between SSTables

Когда RocksDB reads block from SST, он cached в **Block Cache** (LRU). Это shared memory pool, не per-SST. В Flink:

```yaml
state.backend.rocksdb.block.cache-size: 256mb  # per-slot или shared?
state.backend.rocksdb.memory.managed: true     # shared между slots
```

Cache hits — micros. Misses — disk read (NVMe ~50μs, HDD ~5ms, S3 ~50ms).

## Levels и compaction

L0 — special level. Может содержать **overlapping** SST: разные SST могут иметь одинаковые keys (потому что они dumped из MemTable в разные моменты). Reads на L0 должны check **все** SST.

L1+ — каждый level содержит **non-overlapping** SST (если leveled compaction). Reads читают max 1 SST per level.

### Compaction process

Когда `l0_file_num_compaction_trigger` (default 4) достигнут:

1. Background thread picks несколько L0 SST + overlapping L1 SST
2. Merges them (sorted merge join по key)
3. Writes new L1 SST (могут быть больше)
4. Deletes old L0/L1 SST

Это **write amplification** — данные многократно переписываются с level на level. Counterpoint — это что упорядочивает данные для быстрого read.

### Tunable parameters

```yaml
# L0 trigger:
state.backend.rocksdb.level0_file_num_compaction_trigger: 4

# Level size growth:
state.backend.rocksdb.max_bytes_for_level_base: 256MB
state.backend.rocksdb.max_bytes_for_level_multiplier: 10

# Compression per level:
state.backend.rocksdb.compression_per_level:
  - none      # L0: no compression for fast flush
  - none      # L1
  - snappy    # L2+
  - snappy
  - snappy
  - zstd      # cold levels: aggressive
  - zstd
```

### Bloom Filter — критично для performance

Без Bloom Filter каждый GET для несуществующего key читал бы **все** SST на дисках. С Bloom (default enabled):
- 1% false positive rate
- Per-SST in-memory structure (~1-2 bytes per key)
- Read miss skips ~99% SST без disk I/O

Параметры:
```yaml
state.backend.rocksdb.bloom_bits_per_key: 10  # default 10, higher = less false positives
state.backend.rocksdb.use_bloom_filter: true   # critical для read perf
state.backend.rocksdb.cache_index_and_filter_blocks: true  # Block cache
```

## Tombstones и DELETE

DELETE в LSM — это **append** записи tombstone. Original key+value остаётся в SST до compaction. Это означает:

- DELETE = тот же cost что PUT
- Space временно doubles (original + tombstone)
- Read должен check tombstone — если найден, return null
- Compaction physically removes tombstone+original

Для Flink этим важно при state.clear() heavy workloads — tombstones накапливаются, требуют compaction.

```yaml
state.backend.rocksdb.compaction.delete_obsolete_files_period: 6h
```

## Write/Read/Space amplification — fundamental tradeoff

LSM имеет три «amplification» метрики, которые торгуются друг с другом:

### Write Amplification (WA)

`bytes_written_to_disk / bytes_written_by_user`. Каждый byte user-data пишется multiple times при compaction.

Leveled compaction:
```
L0 → L1 (read L0+L1, write merged L1)         ~10x
L1 → L2                                        ~10x
L2 → L3                                        ~10x
...
Total WA для 7 levels: ~50-70x
```

То есть 1GB user writes = 50-70GB disk writes за lifetime. Killing SSD endurance.

### Read Amplification (RA)

`disk_reads_per_user_read`. Сколько раз обращаемся к SST для одного GET.

Leveled: 1 (L0 best case) до 7 (worst case, key on every level). Bloom filter снижает RA для existing keys (1-2 SST) и для missing keys (~0.07).

### Space Amplification (SA)

`disk_space_used / live_data`. Сколько временно дубликатов из-за uncompacted data + tombstones.

Leveled: typically 1.1-1.5 (10-50% overhead).
Universal: 1.5-2 (50-100% overhead).

### The triangle

```
       Write Amp
         /\
        /  \
       /    \
      / LSM  \
     / TRIANGLE\
    /          \
   /────────────\
Read Amp    Space Amp
```

**You can pick two**. Leveled compaction = low SA + low RA + high WA. Universal compaction = lower WA + higher SA. Урок 04 разбирает trade-offs.

## RocksDB tuning knobs (preview)

Основные параметры, влияющие на LSM behaviour:

```yaml
# Memory:
state.backend.rocksdb.writebuffer.size: 64mb
state.backend.rocksdb.writebuffer.count: 2

# Compaction:
state.backend.rocksdb.compaction.style: LEVEL   # или UNIVERSAL
state.backend.rocksdb.compaction.level.use-dynamic-size: true
state.backend.rocksdb.compaction.level.max-size-level-base: 256mb
state.backend.rocksdb.compaction.level.target-file-size-base: 64mb

# Reads:
state.backend.rocksdb.thread.num: 4
state.backend.rocksdb.use-bloom-filter: true
state.backend.rocksdb.block.cache-size: 256mb

# Predefined (recommended starting point):
state.backend.rocksdb.predefined-options: FLASH_SSD_OPTIMIZED
```

## Production gotchas

### 1. L0 file pile-up = write stall

```
L0 files: 24 / max 36 trigger stall
```

Если flush rate > compaction rate, L0 файлы накапливаются. После `level0_slowdown_writes_trigger` (default 20) — RocksDB ARTIFICIALLY slows writes. После `level0_stop_writes_trigger` (default 36) — STOPS writes полностью, ждёт compaction. Это видно в Flink Web UI как полный stall.

**Fix**: больше compaction threads, увеличить L1 size, switch на universal compaction для write-heavy.

### 2. Bloom filter disabled — read perf cliff

Some custom RocksDB configs disable bloom filter. Reads без bloom = O(L * SST_per_L) disk seeks per GET. На large state это 100ms+ per read.

**Always enable bloom**.

### 3. Block cache size mis-set

Слишком маленький (16MB) на TBs state → 99% cache miss → disk-bound. Слишком большой (16GB) на 1GB state → wasted memory.

**Right size**: 10-30% of working set (hot data). Monitor cache hit rate.

### 4. Compression on L0

```yaml
compression_per_level:
  - snappy    # ❌ slows flush
```

Compression на L0 slows MemTable flush — увеличивает risk write stall. Recommended: L0/L1 без compression, L2+ с snappy/zstd.

### 5. Too many SST files

`open files` system limit (`ulimit -n`). Если number SST > limit → file open errors. С 1TB state и 64MB SST per file = 16k files. Default `ulimit -n 1024` insufficient.

Fix: `ulimit -n 65536` в container/systemd.

### 6. Compaction throttling

```yaml
state.backend.rocksdb.thread.num: 1   # ❌ slow compaction
```

Один compaction thread на TBs state = constant L0 pile-up. Set `thread.num: 4-8` для production.

### 7. Sequential vs random key distribution

Flink keys (часто Long, String через MurmurHash) уже random distributed. RocksDB LSM optimized для both, но certain optimizations (key prefix iterator) работают только при sequential prefixes. Обычно не tuneable из Flink.

## Performance numbers (NVMe SSD, 1TB state)

| Operation | Latency P50 | Latency P99 |
|---|---|---|
| MemTable hit (read) | &lt;1μs | 5μs |
| Block cache hit | 10μs | 50μs |
| L0 disk read (NVMe) | 50μs | 200μs |
| L1-L3 disk read | 100μs | 500μs |
| L4-L6 cold disk read | 500μs | 2ms |
| Write (MemTable append) | 5μs | 50μs |
| L0→L1 compaction (64MB files) | 100ms | 500ms |
| Full GC pause (Java side) | 50ms | 500ms |

Compare HDD: ~10x slower. S3 (ForSt without cache): ~100x slower.

## Cheat sheet

```
LSM-tree =
  in-memory MemTable
    + flushed to L0 SSTable (overlapping)
    + compacted into L1+ (non-overlapping, 10x size each level)

Write:    O(log n) to MemTable, almost free
Read:     bloom filter check + up to L disk reads
Tombstone: DELETE = append marker, removed at compaction

Triangle: low Write Amp + low Read Amp + low Space Amp — pick 2
```
