# Application mode vs Session mode

В Flink 2.x остались два deployment-режима: **Application mode** (одна job = один cluster) и **Session mode** (shared cluster для multiple jobs). Третий, исторический режим — **Per-job mode** — **удалён в 2.0** (FLIP-419). Это структурное упрощение: per-job был «не fish, не fowl» — занимал middle ground между двумя другими и приводил к confusion и subtle bugs.

В этом уроке разберём оба режима, lifecycle, isolation guarantees, и когда выбирать какой.

---

## Краткая ретроспектива: что было до 2.0

В Flink 1.x было **три** режима:

| Mode | Cluster lifecycle | main() runs где | Изоляция |
|---|---|---|---|
| **Session** | Long-running cluster, multiple jobs | На клиенте (`flink run`) | Слабая (jobs share TM-ы) |
| **Per-job** | New cluster per job | На клиенте | Сильная (cluster dedicated) |
| **Application** | New cluster per application | **В JobManager-е** | Сильная + main() в кластере |

Различие per-job vs application было тонким: оба создавали dedicated cluster, но per-job компилировал main() на клиенте (медленно для больших JARs), а application — внутри JobManager-а (быстро, и client можно отключить после submit).

**В 2.0 per-job mode удалён** — application полностью его заменяет. Если вы видите legacy doc / config с `cluster.mode: per-job` — это уже не работает.

---

## Application mode: production default

В application mode **main() выполняется в JobManager-е**. Это значит:

1. Клиент upload-ит JAR и triggerит запуск.
2. JobManager pod стартует.
3. **Внутри JobManager** запускается main() — компилирует StreamGraph → JobGraph.
4. JobMaster берёт JobGraph и deploy-ит tasks.
5. Когда job заканчивается — весь cluster shutdown.

### Конфигурация на K8s

```yaml
# В Flink Kubernetes Operator manifest
apiVersion: flink.apache.org/v1beta1
kind: FlinkDeployment
metadata:
  name: my-streaming-app
spec:
  image: flink:2.2.0-java17
  flinkVersion: v2_2
  flinkConfiguration:
    state.backend.type: forst
    state.checkpoints.dir: s3://my-bucket/checkpoints/my-app
    execution.checkpointing.interval: 60s
    high-availability.type: kubernetes
    high-availability.storageDir: s3://my-bucket/ha/my-app
  jobManager:
    resource:
      cpu: 2
      memory: 4096m
  taskManager:
    resource:
      cpu: 4
      memory: 8192m
  job:
    jarURI: local:///opt/flink/usrlib/my-app.jar
    parallelism: 8
    upgradeMode: savepoint
    state: running
```

`local:///` — JAR уже встроен в Docker image. Можно также S3 URI — JM скачает JAR в pod при старте.

### Преимущества

- **Isolation** — каждая application получает свой dedicated JM + TMs. Resource quotas через K8s namespace / resource limits. Один job не «крадёт» slot у другого.
- **Lifecycle clean** — job FINISHED → cluster teardown. Никаких zombie clusters.
- **Simpler ops** — каждая application = один FlinkDeployment manifest. Скоро по числу manifests видно количество прод jobs.
- **main() в JobManager** — компиляция не зависит от client environment. Можно стартовать через CI/CD pipeline без Flink клиента.

### Недостатки

- **Overhead per job** — каждая application имеет свой JM pod (memory, CPU). Для 1000 small jobs это 1000 JM-pods × 2 GB = 2 TB JM memory. Не подходит для many small jobs.
- **Cold start** — JM pod нужно create + JVM startup (10-60 сек). При CI/CD deploys это заметно.
- **Один JobGraph per application** — если в main() вы submit-ите несколько jobs (`env.executeAsync()` несколько раз), cluster будет жить пока все не завершатся, но это discouraged pattern.

## Session mode: shared cluster

В session mode cluster уже запущен, и клиент посылает submission через REST. Cluster обслуживает **многие jobs одновременно**.

### Конфигурация на K8s

```yaml
# Session cluster
apiVersion: flink.apache.org/v1beta1
kind: FlinkDeployment
metadata:
  name: my-session-cluster
spec:
  image: flink:2.2.0-java17
  flinkVersion: v2_2
  flinkConfiguration:
    taskmanager.numberOfTaskSlots: 4
  jobManager:
    resource: { cpu: 2, memory: 4096m }
  taskManager:
    resource: { cpu: 4, memory: 8192m }
    replicas: 5    # 5 TM × 4 slots = 20 slots в pool

---
# Submit jobs в этот cluster
apiVersion: flink.apache.org/v1beta1
kind: FlinkSessionJob
metadata:
  name: my-job-1
spec:
  deploymentName: my-session-cluster
  job:
    jarURI: https://my-bucket.s3.amazonaws.com/job1.jar
    parallelism: 4
```

### Преимущества

- **Resource sharing** — много jobs делят cluster, лучше utilization.
- **Fast submit** — JM уже запущен, job submit за 1-2 сек.
- **Ad-hoc workflows** — отлично для interactive exploration, SQL CLI submissions.

### Недостатки

- **Weak isolation** — heavy job может голодать соседей по CPU/RocksDB cache. Один TM crash убивает slots всех jobs на нём.
- **Failure blast radius** — JM crash убивает все running jobs (recovery через HA, но still все одновременно).
- **Versioning** — все jobs используют один Flink версии (cluster). Нельзя smoothly migrate one job на новую версию без отдельного cluster-а.
- **Single point of contention** — Dispatcher REST endpoint становится bottleneck при много submissions/sec.

## Comparison

| Аспект | Application mode | Session mode |
|---|---|---|
| Cluster lifecycle | Per application | Long-running |
| main() runs где | Inside JM pod | На клиенте (`flink run`) |
| Resource isolation | Hard (dedicated JM + TM) | Soft (shared slots) |
| Failure blast radius | Только эта app | Все jobs в cluster |
| Cold start | 10-60 сек (pod create) | &lt;2 сек (slot allocate) |
| Memory overhead per job | Высокий (свой JM) | Низкий (shared JM) |
| Production use case | Streaming jobs 24/7 | Ad-hoc SQL, exploration, small batch |
| Versioning | Per-app independent | One Flink version per cluster |
| CI/CD integration | Native (one FlinkDeployment YAML) | Через REST API submissions |
| Flink 2.x default? | **Yes** | Legacy / specific use cases |

## Когда какой mode

### Application mode — выбор по умолчанию

- Production streaming jobs (Kafka → process → Kafka, 24/7).
- Каждая job критична и заслуживает dedicated resources.
- CI/CD deployment через Flink Kubernetes Operator.
- Нужна изоляция: SLA / security / compliance.
- Job large state (>100 GB) — dedicated TM нужен полностью.

### Session mode — специфичные cases

- **SQL CLI / ad-hoc queries** — много мелких queries, нет смысла поднимать cluster per query.
- **Interactive notebooks** — Zeppelin / Jupyter с Flink SQL.
- **Many small jobs** — 100+ jobs каждая parallelism=1, slot-share между ними.
- **Development environment** — один dev cluster, на котором разработчики submit-ят jobs из IDE.
- **Stateless ETL fleet** — много stateless transforms, которые можно бесопасно share resources.

<Callout type="warning">
**Anti-pattern**: один production session cluster с десятками stateful jobs. При сбое одного oncall борется одновременно с tens of jobs, восстановление дольше. Лучше — N application clusters с isolated FlinkDeployment manifests.
</Callout>

## Production gotchas

- **Application mode + many small jobs** — overhead JM (~2 GB) делает sub-1-cpu jobs неэкономичными. Для many small jobs — Session mode (или объединение в один job).
- **Session mode + heterogeneous workloads** — heavy state job рядом с CPU-bound. Разные resource needs — конкуренция за managed memory и CPU. Используйте slot sharing groups для isolation, но это не панацея.
- **Application + jobgraph hot reload** — обновление jar означает new cluster (savepoint → terminate → new deploy). Через Flink Kubernetes Operator: `upgradeMode: savepoint` автоматизирует это.
- **Session + crashed JM** — все active jobs восстанавливаются параллельно. Если у вас 50 jobs со state TBs, recovery может занять минуты. Application mode распределяет risk.
- **Per-job mode removed** — если у вас 1.x legacy code/scripts с `cluster.mode: per-job` — мигрируйте на application mode. Documentation для 1.x на этой теме deprecated.
- **client.timeout vs job duration** — в session mode `flink run` без `-d` (detach) ждёт завершения job на клиенте. Для long-running streaming jobs ОБЯЗАТЕЛЬНО `-d`, иначе client-side timeout прервёт submission.

<Callout type="tip">
Mature production setup на K8s: **Application mode + Flink Kubernetes Operator + FlinkDeployment per job + GitOps**. Каждая job в репо как YAML, ArgoCD/Flux синхронизируют изменения с cluster, savepoint-based upgrades через `upgradeMode: savepoint`. Session mode только для dev/staging.
</Callout>
