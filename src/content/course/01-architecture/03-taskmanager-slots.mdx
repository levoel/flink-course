# TaskManager и слоты: memory model FLIP-49

TaskManager — worker-процесс Flink, JVM с N task slots. Slot — единица планирования: каждый subtask занимает один slot. Это звучит просто, но за фразой «slot — единица планирования» скрывается тонкая модель: что именно изолируется между slots, что — нет, как делится память, что произойдёт, если поставить `numberOfTaskSlots=8` на TM с 2 ГБ heap.

В этом уроке препарируем TaskManager на компоненты, разберём memory model FLIP-49 (active since Flink 1.10), и поймём, как RocksDB-state, network buffers и user heap делят process memory.

---

## TaskManager как JVM-контейнер

TaskManager — это **одна JVM**, которая хостит N subtasks разных jobs (в Session mode) или N subtasks одной job (в Application mode). Каждый subtask — это не отдельный поток в чистом виде, а целая «mailbox» с одним worker-thread и набором ресурсов.

**Ключевая идея**: slot — это не CPU core и не отдельный thread pool. Slot — это **memory budget + identifier для slot-aware scheduling**. CPU делится OS-планировщиком между всеми потоками JVM, network buffers общие, RocksDB block cache общий через WriteBufferManager (FLIP-49).

### Пример распределения 4 ГБ

```yaml
taskmanager.memory.process.size: 4096m       # 4 GB total
taskmanager.memory.framework.heap.size: 128m  # 128 MB
taskmanager.memory.framework.off-heap.size: 128m  # 128 MB
taskmanager.memory.jvm-metaspace.size: 256m   # 256 MB
taskmanager.memory.jvm-overhead.fraction: 0.1  # 409 MB (clamped to range)
taskmanager.memory.network.fraction: 0.1       # 409 MB
taskmanager.memory.managed.fraction: 0.4       # 1638 MB (RocksDB!)
# Task heap = 4096 - 128 - 128 - 256 - 409 - 409 - 1638 = 1128 MB
```

То есть из 4 ГБ только ~1.1 ГБ доступно user heap. Если у вас `numberOfTaskSlots: 4`, на slot приходится ~280 MB heap. Это часто меньше, чем ожидают разработчики.

<Callout type="warning">
Самая частая ошибка — поднять `numberOfTaskSlots` без увеличения `process.size` и удивиться OOM. Каждый slot уменьшает heap budget per subtask. На 4 GB TM с 8 slots на subtask приходится &lt;150 MB — нереалистично для большинства workloads.
</Callout>

## Что shared между slots, что — нет

| Ресурс | Изоляция между slots | Комментарий |
|---|---|---|
| **Task Heap** | Soft (общий JVM heap, но логически делится) | GC видит весь heap как один. OOM в одном slot убьёт всю TM. |
| **Managed Memory** | Hard (через MemoryManager) | RocksDB через WriteBufferManager делит block cache. Каждый slot получает managed/N байт квоту. |
| **Network Buffers** | Hard (через NetworkBufferPool) | Каждый channel получает свой набор buffers. |
| **CPU** | None (soft isolation) | OS-планировщик распределяет потоки. Heavy CPU job голодает соседей. Для hard CPU isolation — отдельные TM-ы. |
| **RocksDB block cache** | Hard (shared cache through MemoryManager) | Включена по умолчанию (state.backend.rocksdb.memory.managed: true) — все instances одного TM делят cache. |
| **JVM Metaspace** | None (общий JVM) | Загруженные классы разделяются между slots. |
| **Threads** | Logical (mailbox thread per subtask) | Каждый subtask имеет ровно один worker thread (mailbox processor). |

Это значит: **slot изолирует state, не CPU**. Если на одном TM два slot-а, один из которых делает heavy CPU computation, второй замедлится. Для строгой CPU isolation — поднимайте больше TM-ов с одним slot каждый (на K8s — pod-per-slot).

## Mailbox processor — модель потоков

Каждый subtask имеет ровно один поток — **mailbox thread**. Это не thread pool, не reactor — это classic single-threaded event loop:

```
loop {
  if (timer expired) onTimer();
  else if (input available) processElement();
  else if (mail available) processMail();   // checkpoints, watermarks, control
  else park();
}
```

Это устраняет race conditions внутри subtask. State доступ без локов, timers выполняются sequentially с processElement, checkpoints (barriers) обрабатываются как специальные «mail» сообщения.

**Side effect**: блокирующая операция в `processElement` (synchronous DB call, slow disk read) останавливает весь mailbox — checkpoints не успевают, watermarks не propagate. Поэтому для external lookups обязателен Async I/O (модуль 02 урок 06).

## Сколько slots ставить

Эвристика для production:

```
desired_total_slots = sum of parallelism всех jobs
slots_per_tm = numberOfTaskSlots
total_tms = desired_total_slots / slots_per_tm
process.size_per_tm = N * memory_per_slot + framework_overhead
```

Типичные настройки:

| Workload | Slots per TM | Process size per TM | Комментарий |
|---|---|---|---|
| **Stateless transforms** | 4-8 | 2-4 GB | Heap-heavy, не нужен большой managed budget. |
| **Heavy state (RocksDB)** | 1-2 | 8-16 GB | Managed memory нужно ≥ 4 GB для приличного block cache. |
| **CPU-intensive (parsing, ML inference)** | 1 | 4-8 GB | Hard CPU isolation через separate pods. |
| **Mixed workload** | 2-4 | 4-8 GB | Компромисс. |

## Comparison vs Spark / Kafka Streams

| Аспект | Flink TaskManager | Spark Executor | Kafka Streams |
|---|---|---|---|
| Process model | Standalone JVM, registered с RM | Standalone JVM, registered с driver | Embedded в application JVM |
| Unit of scheduling | Slot | Core (cores per executor) | Stream thread |
| Memory isolation | Hard via MemoryManager (managed/network) | Heap + off-heap, не isolated per task | Shared с application |
| CPU isolation | Soft (OS scheduler) | Soft (OS scheduler) | Soft |
| State storage | RocksDB / ForSt managed memory | In RDD between batches | RocksDB embedded |
| Multiple jobs per worker | Yes (Session mode) | No (executor per app) | Multiple apps possible но independent |

## Production gotchas

- **`numberOfTaskSlots: 1` + много TM-ов** — даёт hard CPU isolation, но увеличивает network overhead (больше shuffle между процессами). Хороший выбор для state-heavy / CPU-heavy workloads.
- **`numberOfTaskSlots: N` + один TM** — экономит память (один framework heap, одна Java metaspace), но soft isolation. Хорошо для stateless transforms.
- **OOM-Killed pod** — почти всегда означает, что native memory (RocksDB / Netty direct) превысила процесс лимит. Включайте managed RocksDB (`state.backend.rocksdb.memory.managed: true` — default в 2.x), проверяйте, что custom OptionsFactory не отключает WriteBufferManager.
- **Heap dump показывает много `byte[]` без owner** — это network buffers + RocksDB. Не паника, это by design.
- **Long GC pauses (>5s)** — обычно managed/heap соотношение неверное. Увеличьте managed (там RocksDB) и уменьшайте heap. На больших heap (>16 GB) переходите на G1 или ZGC.
- **`jvm-overhead` clamp** — fraction 0.1 при 4 GB process даёт 409 MB, но при 200 GB даёт 20 GB, что perekruchivaet min/max (default min 192MB max 1GB). Проверяйте actual через TM metrics.

<Callout type="tip">
Web UI → TaskManager → Memory tab показывает realtime распределение. Если managed используется ~90%+ — добавьте `managed.fraction` или поднимите `process.size`. Если task heap close to max — у вас leak в user code или slishком много slots.
</Callout>
