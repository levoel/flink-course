# JobManager: Dispatcher, ResourceManager, JobMaster

JobManager — это не один сервис, а **три внутренних компонента** в одном JVM-процессе, каждый со своей зоной ответственности и жизненным циклом. Эта декомпозиция — не academic detail, а ключ к пониманию HA, application mode и того, что произойдёт, когда умрёт лидер кластера.

В этом уроке препарируем JobManager на Dispatcher (REST-фасад кластера), ResourceManager (распределение slots), JobMaster (lifecycle per job), и разберём HA через ZooKeeper и Kubernetes ConfigMap.

---

## Зачем три компонента, а не один

Изначально (Flink 1.0–1.4) JobManager был монолитом. С 1.5 (FLIP-6) его разделили — это позволило поддерживать одновременно multiple jobs, гибко эмбедить ResourceManager под разные cluster managers (Standalone / YARN / K8s), и сделать HA semantics чище: при leader change новый Dispatcher восстанавливает submitted jobs из persistent storage, ResourceManager поднимается отдельно, JobMaster каждой job восстанавливается из последнего checkpoint.

## Dispatcher: фасад кластера

Dispatcher — то, с чем общается `flink` CLI и Web UI. Он принимает JobGraph (сериализованный результат компиляции вашего main-метода) и инициирует его исполнение. Важные обязанности:

1. **Приём submissions** — REST `/jars/upload` загружает JAR, `/jars/{jarid}/run` компилирует main → JobGraph → запускает.
2. **Persistence** — JobGraph пишется в `high-availability.storageDir` (S3/HDFS) с metadata-пойнтером в ZK/K8s ConfigMap. Если Dispatcher умирает, новый лидер читает JobGraph из storage и поднимает JobMaster заново.
3. **Cluster-wide REST** — `/jobs`, `/jobs/{id}/checkpoints`, `/jobs/{id}/savepoints`, `/jobs/{id}/yarn-cancel` и так далее.
4. **Job retention** — после завершения job (FINISHED/CANCELED/FAILED) Dispatcher хранит execution graph store до `jobstore.expiration-time` для post-mortem анализа.

Один Dispatcher на cluster (в Application mode — один на cluster и одна job, в Session mode — один на много jobs).

```bash
# Dispatcher endpoints
curl http://jobmanager:8081/overview
curl http://jobmanager:8081/jobs
curl http://jobmanager:8081/jobs/<jobId>/checkpoints
```

## ResourceManager: учёт slots

Один ResourceManager на cluster. Его роль — abstraction над тем, как Flink получает compute-ресурсы:

| Cluster type | Implementation | Поведение |
|---|---|---|
| Standalone | `StandaloneResourceManager` | Регистрирует уже запущенные TaskManager-ы. Не может создавать новые. |
| Kubernetes Native | `KubernetesResourceManager` | Создаёт новые TM-pod-ы через K8s API при нехватке slots. Удаляет idle pod-ы по `slotmanager.taskmanager-timeout`. |
| YARN | `YarnResourceManager` | Запрашивает containers через AM-RM protocol, релизит unused. |

JobMaster при старте просит у ResourceManager-а N slots (по parallelism job-а). Если slots не хватает:

- **Standalone**: job ждёт `slot.request.timeout` (default 5 min), потом FAILED.
- **K8s Native / YARN**: ResourceManager инициирует pending allocation, поднимает дополнительные TaskManager-ы.

```yaml
# Adaptive Scheduler + reactive K8s
jobmanager.scheduler: adaptive
slot.request.timeout: 5min
slotmanager.taskmanager-timeout: 30s     # idle TM удаляется через 30с
kubernetes.taskmanager.cpu: 2.0
kubernetes.taskmanager.memory: 4096m
```

ResourceManager также управляет **slot pool** per-JobMaster — кэш предложенных slots, чтобы при minor failure не запрашивать заново.

<Callout type="note">
В Application mode JobManager pod-а имеет один Dispatcher, один ResourceManager и один JobMaster (для единственной job этой application). В Session mode тот же Dispatcher и ResourceManager обслуживают много параллельных JobMaster-ов.
</Callout>

В обычном lifecycle JobMaster проходит фазы: CREATED → RUNNING → (CANCELLING|FAILING) → (FINISHED|CANCELED|FAILED). RUNNING — состояние, в котором JobMaster ждёт сигналов от tasks и периодически триггерит checkpoints. Деплой tasks на TaskManager-ы JobMaster делает через RPC `TaskExecutorGateway.submitTask(TaskDeploymentDescriptor)` — TDD содержит сериализованный код task-а, его inputs/outputs и slot id.

## HA через ZooKeeper и Kubernetes ConfigMap

Без HA Services рестарт JobManager означает потерю всех submitted jobs (Dispatcher не помнит JobGraphs). HA Services решают это через persistent state в ZK или K8s.

### ZooKeeperHaServices

Используется в bare-metal / VM / Hadoop окружениях:

```yaml
high-availability.type: zookeeper
high-availability.zookeeper.quorum: zk1:2181,zk2:2181,zk3:2181
high-availability.zookeeper.path.root: /flink
high-availability.storageDir: s3://my-bucket/flink-ha
high-availability.cluster-id: production-cluster
```

ZK хранит **только pointers**: пути к JobGraph blob-ам в S3, идентификатор текущего leader-а, latest checkpoint pointer. Большие объекты — в `storageDir`. Это позволяет ZK-ноды держать малыми (znode limit 1MB).

### KubernetesHaServices

С Flink 1.12+ HA можно делать без ZK через K8s native ConfigMaps:

```yaml
high-availability.type: kubernetes
high-availability.storageDir: s3://my-bucket/flink-ha
kubernetes.cluster-id: production-cluster
```

K8s ConfigMap содержит leader info + JobGraph pointers, persistent storage в S3 для blob-ов. Leader election — через K8s lease mechanism. Это снимает зависимость от ZK для cloud-native deployments — типичный setup для Flink Kubernetes Operator.

### Leader election

При старте каждый JobManager pod пытается стать лидером:

1. **ZK**: запись в ephemeral znode `/leader/dispatcher` с lock. Кто первый — тот лидер. При смерти владельца ephemeral znode исчезает, и кто-то другой триггерит resubscribe и захватывает lock.
2. **K8s**: каждый pod пытается update ConfigMap с annotation `leader=<pod-id>` через optimistic lock. Lease (renew раз в N секунд) гарантирует, что dead pod не блокирует election.

Leader election происходит отдельно для каждого сервиса: Dispatcher leader, ResourceManager leader, per-JobMaster leader. Обычно они на одном pod-е, но логически независимы.

## Что значит «JobManager умер»

| Сценарий | Что происходит |
|---|---|
| **Без HA** | Все running jobs FAIL. State lost (если checkpoint dir не на DFS). Cluster нужно поднимать руками. |
| **С HA, ZK/K8s leader change** | Standby JM становится leader через 5-20 сек. Восстанавливает JobGraphs из storage. JobMaster каждой job стартует, читает latest checkpoint, deploy-ит tasks на оставшиеся TM-ы. TM-ы переключают connection на новый JM. |
| **TaskManager умер** | JobMaster детектит через heartbeat timeout (default 50s). Все subtasks на этом TM перешли в FAILED. Failover triggered — restart pipelined region. State recovered из last checkpoint. |
| **Dispatcher умер, JobMaster жив** | Dispatcher не critical для running jobs — JobMaster продолжает работать, общается с ResourceManager напрямую. Новые submissions блокируются до восстановления Dispatcher. |

<Callout type="warning">
**Важный gotcha**: heartbeat timeout default 50 сек — это значит при transient network glitch (10-30 сек) job выживает, но TM считается мёртвым после 50с. На K8s с graceful termination и rolling updates стоит уменьшить `heartbeat.timeout` до 30 секунд, иначе rolling restart TM-ов вызовет десятки restart-ов pipeline.
</Callout>

## Production gotchas

- **Dispatcher не масштабируется** — один сервис на cluster. Если у вас 1000+ jobs в Session mode, узким местом будет Dispatcher REST endpoint, не TaskManagers. Решение — Application mode или несколько Session clusters.
- **ResourceManager на K8s медленный** — поднятие нового TM-pod-а занимает 10-60 секунд (image pull, JVM startup). Если parallelism job > available slots, job может ждать минуты. Pre-warm TM-ов через `kubernetes.jobmanager.replicas` + `slotmanager.number-of-slots.min`.
- **Cluster-id collision** — два разных Flink cluster с одним `high-availability.cluster-id` в одном ZK/K8s namespace будут конфликтовать за leadership и потеряют state. Всегда уникальный `cluster-id` per environment.
- **HA storage permissions** — JobManager pod должен иметь write access к `storageDir`. На K8s это IRSA / Workload Identity, не credentials в env. После recovery права на чтение existing blob-ов критичны.
