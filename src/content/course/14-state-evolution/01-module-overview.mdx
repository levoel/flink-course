# State Evolution — обзор модуля

Production Flink job живёт годами. За это время меняется: schema state, topology, backend, parallelism, version Flink. Этот модуль учит, как мигрировать state safely.

## State Processor API — superpower

Read/write state offline:

```java
SavepointReader sp = SavepointReader.read(env, savepointPath);

DataStream<KeyedState> keyed = sp.readKeyedState(
    "fraud-detector-uid",
    new ReaderFunction()
);

// Modify, filter, aggregate state

SavepointWriter newSp = SavepointWriter.fromExistingSavepoint(savepointPath)
    .removeOperator("old-uid")
    .withOperator("new-uid", new BootstrapTransformation<>(...));

newSp.write(newSavepointPath);
```

## Savepoint formats

| Format | Speed | Cross-backend | When |
|---|---|---|---|
| **Native** (default 1.15+) | Fast | ❌ | Same backend, frequent updates |
| **Canonical** | Slow | ✅ | Backend migration (RocksDB → ForSt) |

```bash
flink stop --savepoint-format canonical --savepointPath ... <jobid>
```

## UIDs — обязательно

```java
stream
    .keyBy(...)
    .process(new FraudDetector())
    .uid("fraud-detector")    // ← MANDATORY для production
    .name("Fraud Detection");
```

Без UID Flink не может сопоставить state из savepoint с оператором после изменения topology.

## Migration playbook

1. Stop job with **canonical** savepoint
2. Update code / config
3. Run new version с `flink run -s savepoint --allowNonRestoredState`
4. Verify state correctly restored
5. Monitor stability сутки → only then удалить старую savepoint

## Killer takeaway

**Канонический совет**: установите UID на каждый stateful operator + используйте canonical savepoints для cross-backend migrations + регулярно testing rollback. Это спасёт от disaster в production.
