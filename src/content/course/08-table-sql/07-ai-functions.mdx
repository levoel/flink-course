# AI functions — ML_PREDICT и VECTOR_SEARCH (Flink 2.2)

Flink 2.2 (декабрь 2025) принёс две flagship AI features в SQL — **ML_PREDICT** (FLIP-526) для inference ML моделей и **VECTOR_SEARCH** (FLIP-540) для vector similarity search. Это disruptive: ML inference pipeline в 10 строк SQL вместо custom DataStream code с external HTTP calls. Этот урок разбирает CREATE MODEL syntax, providers (OpenAI, custom HTTP endpoints, embedded ONNX), vector indexing strategies и production patterns для real-time ML pipelines.

Это не just convenience — это новая category streaming applications: real-time fraud scoring через GPT, semantic search streaming results, embedding-based recommendation в дашборде.

---

## Контекст — почему AI в streaming SQL

Без AI functions ML inference в Flink требовал:
1. AsyncFunction с HTTP client → OpenAI API.
2. Manual rate limiting, retry, batching.
3. Custom serialization запроса/ответа.
4. Error handling per call.
5. No deterministic replay (HTTP responses vary).

С ML_PREDICT:
```sql
SELECT user_id,
       ML_PREDICT('fraud_classifier', transaction) AS fraud_score
FROM transactions;
```

Под капотом — Async I/O, retry, caching, batching. Declarative.

## CREATE MODEL — register inference endpoint

```sql
CREATE MODEL fraud_classifier
INPUT (transaction ROW<user_id STRING, amount DECIMAL, location STRING>)
OUTPUT (score DOUBLE, reasoning STRING)
WITH (
    'provider' = 'openai',
    'task' = 'completion',
    'endpoint' = 'https://api.openai.com/v1/chat/completions',
    'model' = 'gpt-4',
    'system_prompt' = 'You are a fraud detection assistant. Analyze the transaction and return JSON {"score": 0-1, "reasoning": "..."}',
    'api_key' = 'sk-...',  -- или через env: '${OPENAI_API_KEY}'
    'temperature' = '0.1',
    'max_tokens' = '500',
    'rate.limit.rpm' = '100',
    'batch.size' = '10'
);
```

Concepts:
- **INPUT** — schema of input feature row.
- **OUTPUT** — schema of model output. Flink parses response к этой schema.
- **provider** — OpenAI, custom HTTP, ONNX embedded, TensorFlow Serving.
- **task** — completion, embedding, classification.
- **rate/batch settings** — built-in throttling и batching.

## ML_PREDICT — inline inference

```sql
SELECT
    t.tx_id,
    t.user_id,
    t.amount,
    pred.score AS fraud_score,
    pred.reasoning AS fraud_reasoning
FROM transactions AS t,
LATERAL TABLE(ML_PREDICT(MODEL fraud_classifier, t.* )) AS pred;
```

`LATERAL TABLE(ML_PREDICT(...))` — table function expansion. Per input row, model inference produces output row(s), joined as additional columns.

### Simpler syntax (function form)

```sql
SELECT
    tx_id,
    user_id,
    ML_PREDICT('fraud_classifier', tx_id, user_id, amount, location) AS prediction
FROM transactions;
```

Returns single ROW value, can access fields:
```sql
SELECT
    tx_id,
    prediction.score AS score,
    prediction.reasoning AS reasoning
FROM ...;
```

### Async behavior

Under the hood: Async I/O запрос к provider. Default queue capacity 100 (configurable через model option). Throughput limited by API rate limit + latency.

Performance estimates:
- OpenAI GPT-4: ~1-5 sec latency → ~20-50 inferences/sec per worker.
- Custom local ONNX: ~10-100 ms → ~100-1000/sec.
- Embedded model (FLINK_AI): batched, ~10K/sec.

## VECTOR_SEARCH — similarity search в SQL

Vector embeddings + similarity search built-in.

### Create vector index

```sql
CREATE TABLE products (
    product_id STRING,
    name STRING,
    description STRING,
    embedding ARRAY<FLOAT>,  -- precomputed embedding vector (e.g., 1536-dim)
    PRIMARY KEY (product_id) NOT ENFORCED
) WITH (
    'connector' = 'paimon',
    'vector.column' = 'embedding',
    'vector.index.type' = 'hnsw',           -- or 'ivf', 'flat'
    'vector.index.metric' = 'cosine',       -- or 'l2', 'dot_product'
    'vector.index.dim' = '1536'
);
```

Paimon stores vectors with HNSW index. Other connectors: Pinecone, Weaviate, Milvus — through specific connector.

### Query

```sql
SELECT
    o.order_id,
    o.search_query,
    similar.product_id,
    similar.name,
    similar.similarity_score
FROM orders AS o,
LATERAL TABLE(VECTOR_SEARCH(
    TABLE products,
    'embedding',                              -- index column
    o.search_embedding,                       -- query vector
    5,                                        -- top-K
    'cosine'                                  -- distance metric
)) AS similar(product_id, name, similarity_score);
```

Returns top-K similar products per order's search embedding. Streaming nature: query vector комбинируется с indexed vectors continuously.

### Combine с ML_PREDICT для full RAG

```sql
-- Use case: чат-bot с product recommendation
CREATE MODEL query_embedder
INPUT (query STRING)
OUTPUT (embedding ARRAY<FLOAT>)
WITH (
    'provider' = 'openai',
    'task' = 'embedding',
    'model' = 'text-embedding-3-large',
    'api_key' = '${OPENAI_API_KEY}'
);

SELECT
    q.session_id,
    q.user_query,
    similar.product_id,
    similar.name
FROM user_queries AS q,
LATERAL TABLE(VECTOR_SEARCH(
    TABLE products,
    'embedding',
    -- inline embedding inference!
    ML_PREDICT('query_embedder', q.user_query).embedding,
    3,
    'cosine'
)) AS similar(product_id, name, score);
```

Per user query: embed it (ML_PREDICT), search products (VECTOR_SEARCH), join all in one SQL.

## State / external system trade-offs

Vector index storage:

| Backend | Vector count | Latency P99 | Streaming friendly |
|---|---|---|---|
| Paimon с HNSW | 10M-100M | 5-20ms | ✅ native |
| Pinecone (SaaS) | 100M+ | 10-50ms | ✅ через connector |
| Weaviate (self-hosted) | 100M+ | 5-30ms | ✅ |
| Milvus (self-hosted) | 1B+ | 10-100ms | ✅ |
| Flink in-state (HashMap) | 1M-10M | <5ms | ✅ для small |
| External REST API | depends | 50-200ms | ⚠️ через ML_PREDICT |

For most production: Paimon (lightweight) или dedicated vector DB.

## Real-time pipeline examples

### Example 1 — real-time fraud scoring

```sql
CREATE MODEL fraud_scorer
INPUT (features ROW<amount DOUBLE, hour INT, location_risk DOUBLE>)
OUTPUT (score DOUBLE)
WITH (
    'provider' = 'http',
    'endpoint' = 'https://ml-internal.com/fraud',
    ...
);

CREATE MATERIALIZED TABLE fraud_alerts
FRESHNESS = INTERVAL '5' SECOND
AS SELECT
    tx_id,
    user_id,
    amount,
    pred.score AS fraud_score
FROM transactions AS t,
LATERAL TABLE(ML_PREDICT(MODEL fraud_scorer, t.amount, EXTRACT(HOUR FROM t.event_time), t.location_risk)) AS pred
WHERE pred.score > 0.8;
```

Materialized table updates каждые 5 секунд с new fraud alerts. Downstream alerting reads this table.

### Example 2 — Semantic search streaming

```sql
CREATE TABLE search_log (
    session_id STRING,
    query STRING,
    event_time TIMESTAMP(3),
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
) WITH ('connector' = 'kafka', ...);

CREATE MODEL embedder
INPUT (text STRING)
OUTPUT (embedding ARRAY<FLOAT>)
WITH ('provider' = 'openai', 'task' = 'embedding', 'model' = 'text-embedding-3-large', ...);

CREATE MATERIALIZED TABLE search_results
FRESHNESS = INTERVAL '1' SECOND
PRIMARY KEY (session_id, rank) NOT ENFORCED
AS SELECT
    s.session_id,
    s.query,
    ROW_NUMBER() OVER (PARTITION BY s.session_id ORDER BY similar.score DESC) AS rank,
    similar.product_id,
    similar.name,
    similar.score
FROM search_log AS s,
LATERAL TABLE(VECTOR_SEARCH(
    TABLE products,
    'embedding',
    ML_PREDICT('embedder', s.query).embedding,
    10,
    'cosine'
)) AS similar(product_id, name, score);
```

Real-time search-as-you-type backed by streaming embeddings.

### Example 3 — Content moderation

```sql
CREATE MODEL moderator
INPUT (content STRING)
OUTPUT (toxic_score DOUBLE, categories ARRAY<STRING>)
WITH ('provider' = 'openai', 'task' = 'moderation', 'model' = 'omni-moderation-latest', ...);

CREATE MATERIALIZED TABLE moderated_posts
FRESHNESS = INTERVAL '2' SECOND
AS SELECT
    p.post_id,
    p.user_id,
    p.content,
    mod.toxic_score,
    mod.categories,
    CASE WHEN mod.toxic_score > 0.7 THEN 'BLOCK'
         WHEN mod.toxic_score > 0.3 THEN 'REVIEW'
         ELSE 'OK' END AS action
FROM social_posts AS p,
LATERAL TABLE(ML_PREDICT(MODEL moderator, p.content)) AS mod;
```

Real-time moderation pipeline, decision logic в SQL.

## Performance, batching, caching

### Batching automatic

```sql
WITH ('batch.size' = '20')
```

Flink collects up to 20 rows, sends batch request to provider, distributes responses. Reduces API overhead 10-50x для OpenAI.

### Caching

```sql
WITH (
    'cache.enabled' = 'true',
    'cache.max-rows' = '10000',
    'cache.expire-after-write' = '1h'
)
```

LRU cache на model inputs. Identical inputs serve from cache. Massive savings for repetitive inputs.

### Rate limiting

```sql
WITH (
    'rate.limit.rpm' = '500',   -- requests per minute
    'rate.limit.tpm' = '60000', -- tokens per minute (LLMs)
    'rate.limit.strategy' = 'block_until_available'
)
```

Built-in rate limiting matches provider quotas. Avoids 429 errors.

### Async queue size

```sql
WITH ('async.queue.capacity' = '200')
```

How many concurrent inferences. Tune based on expected latency.

## Production gotchas

### 1. No deterministic replay для LLM-based predictions

```sql
ML_PREDICT('gpt-4', ...)
```

Same input → may return different output (especially temperature > 0). Replay не deterministic. State checkpoint can't guarantee exact outputs after replay.

**Fix**: 
- `temperature=0` для maximum determinism.
- Cache responses (cache.enabled=true) — replay hits cache.
- Accept that ML output is non-deterministic, design downstream accordingly.

### 2. API key in DDL plain text

```sql
WITH ('api_key' = 'sk-real-key-12345')
```

DDL stored в catalog. Anyone с catalog access sees key.

**Fix**: env var interpolation `${OPENAI_API_KEY}` (Flink reads from TM env). Or secrets manager integration.

<Callout type="tip">
Cost-aware design для production ML pipelines: two-stage classification — cheap embedded ONNX (DistilBERT) handles 80-90% high-confidence, expensive LLM fallback только для low-confidence (10-20%). Plus cache.enabled=true для repetitive content. Это снижает stocks 100× vs naive «ML_PREDICT every event».
</Callout>

### 3. Rate limit ignored → 429 storm

Without proper rate.limit settings, Flink can exceed provider limits → 429 cascades. Producer back-pressured, no progress.

**Fix**: ALWAYS set rate.limit.rpm matching API tier. Plus monitor 429 rate.

### 4. Vector dimension mismatch

```sql
CREATE TABLE products (embedding ARRAY<FLOAT>) WITH ('vector.index.dim' = '1536');
-- Later inserted vectors с dim 768 — corruption
```

Index built for 1536-dim, inserting 768-dim breaks.

**Fix**: validation pipeline upstream, или DDL constraint check.

### 5. Latency cliff from cold model

First inference after restart: model load, connection setup. Latency P99 may spike.

**Fix**: 
- Embedded models pre-loaded в open().
- Health check before traffic.
- Warm-up phase в pipeline initialization.

### 6. Cost — every event hits API

```sql
SELECT ML_PREDICT('gpt-4', text) FROM events  -- 1M events/day × $0.03/inference = $30K/day
```

Naive ML_PREDICT on high-volume stream costs $$$.

**Fix**: 
- Pre-filter с simple rules before LLM (regex, threshold).
- Cache repetitive inputs.
- Sampling (predict 1 of 10 events).
- Cheaper model для first pass, expensive только если flagged.

### 7. PII в LLM requests

Sending raw user data к OpenAI may violate GDPR/HIPAA. Logs of API contain sensitive info.

**Fix**: 
- Self-hosted model для PII.
- Pseudonymize features before ML_PREDICT.
- Data residency-aware provider (Azure OpenAI с EU region).

### 8. Vector search recall vs latency

```sql
'vector.index.type' = 'hnsw'  -- fast but approximate
```

HNSW returns approximate nearest neighbors — может miss true top-K. For exact: 'flat' (full scan, slow).

**Fix**: tune `ef_search` parameter для recall/latency balance.

```
CREATE MODEL name
INPUT (...)
OUTPUT (...)
WITH (
    'provider' = 'openai | http | anthropic | onnx | tensorflow',
    'model' = '...',
    'api_key' = '${ENV_VAR}',
    'rate.limit.rpm' = 'N',
    'batch.size' = 'N',
    'cache.enabled' = 'true'
);

ML_PREDICT(model_name, input_fields...) → ROW

VECTOR_SEARCH(
    TABLE indexed_table,
    'vector_column',
    query_vector,
    top_k,
    metric: 'cosine' | 'l2' | 'dot_product'
) → table function

Use cases:
    Fraud scoring     — features → score
    Content moderation — text → toxic_score
    Semantic search   — query → embedding → VECTOR_SEARCH
    Recommendation    — user_history → embedding → similar items
    Translation       — text → translated
```
