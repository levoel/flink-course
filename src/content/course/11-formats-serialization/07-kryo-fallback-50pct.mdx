# Kryo fallback — 50% perf hit

Каждый раз, когда Flink не может вывести правильный TypeInformation, он откатывается на `GenericTypeInfo<T>`, который внутри использует Kryo serializer. Kryo — universal Java serializer библиотека, работает для практически любых классов через reflection. Это convenient escape hatch, но в production он **тихо съедает 50% производительности**, ломает schema evolution и приводит к state corruption при upgrade Flink версий.

Этот урок — о том, почему Kryo fallback так разрушителен, как Flink тихо к нему скатывается, и как настроить fail-fast поведение через `pipeline.generic-types: false`. **Каждый production deployment должен иметь этот config.** Если у вас его нет — у вас Kryo fallback где-то в pipeline, вы просто не знаете где.

---

## Что такое Kryo и почему он медленный

Kryo (Esoteric Software) — Java serialization library с фокусом на скорость и compact wire format. По сравнению с Java built-in `ObjectOutputStream`, Kryo в 5-10x быстрее. **Но Kryo медленнее любого schema-aware serializer**, потому что:

1. **Class FQN или ID в wire** — каждая запись пишет либо полное class name, либо short class ID (если registered). FQN — typically 30-50 bytes overhead per record.
2. **Reflection-based field discovery** — Kryo не имеет pre-built schema, проходит fields через reflection per object.
3. **No structural validation** — на deserialize Kryo пытается заполнить fields в whatever class found by FQN. Если class изменился — silent corruption.
4. **Per-instance generic reasoning** — для generic collections Kryo пишет component type FQN per element.

```
POJO baseline:       1 μs per record, 24 bytes
Avro Specific:      1.2 μs per record, 22 bytes
Kryo registered:    1.5 μs per record, 32 bytes  (with short class ID)
Kryo unregistered:  2.5 μs per record, 60+ bytes (with class FQN)
```

В numbers: для 100k events/sec/subtask и 8 parallelism = 800k events/sec total → Kryo cost ~1500 CPU ms/sec vs ~700 ms для POJO. Это **2x CPU** просто на serialization.

## Six common ways к Kryo

### Way 1: Lambda без `.returns()`

```java
DataStream<Tuple2<String, Long>> stream = input
    .map(v -> Tuple2.of(v.userId, 1L));   // ← НЕТ .returns()
// → InvalidTypesException или GenericTypeInfo<Tuple2>
```

Fix:

```java
.map(v -> Tuple2.of(v.userId, 1L))
.returns(Types.TUPLE(Types.STRING, Types.LONG));
```

### Way 2: Lombok без `@NoArgsConstructor`

```java
@Data @Builder    // ← @NoArgsConstructor отсутствует
public class FraudAlert {
    private long alertId;
    private String userId;
    private Instant detectedAt;
}
// → GenericTypeInfo<FraudAlert> → Kryo
```

Fix:

```java
@Data @Builder @NoArgsConstructor @AllArgsConstructor
public class FraudAlert { ... }
```

### Way 3: Non-Flink field type внутри POJO

```java
public class Order {
    public long id;
    public String customer;
    public Instant orderTime;     // ← не POJO type
    public Optional<String> note; // ← не POJO type
    public Order() {}
}
// PojoTypeInfo<Order>, но per-field:
//   id: Long (BasicTypeInfo, fast)
//   customer: String (BasicTypeInfo, fast)
//   orderTime: GenericType<Instant> ← Kryo
//   note: GenericType<Optional> ← Kryo
```

Класс **сам** распознан как POJO, но **fields внутри** — Kryo. Это самый коварный случай: throughput не падает на 50%, а зависит от какие fields hot.

Fix: replace `Instant orderTime` с `long orderTimeMs`, replace `Optional<String> note` с `@Nullable String note`.

### Way 4: Inner non-static class

```java
public class Outer {
    public class BadInner {        // ← non-static
        public long id;
        public BadInner() {}
    }
}
// BadInner имеет synthetic this$0 reference на Outer → не POJO
```

Fix: `public static class GoodNested { ... }`.

### Way 5: Generic POJO

```java
public class Wrapper<T> {
    public T value;
    public long timestamp;
    public Wrapper() {}
}

DataStream<Wrapper<String>> stream = ...;
// T параметр generic — TypeExtractor не может разрешить → Kryo
```

Fix: либо конкретные subclasses (`StringWrapper extends Wrapper<String>`), либо `.returns(new TypeHint<Wrapper<String>>(){})`.

### Way 6: Scala collections / Kotlin Pair

```java
import scala.Tuple2;

DataStream<Tuple2<String, Long>> stream = ...;  // ← Scala Tuple, не Flink Tuple
// → Kryo path
```

```kotlin
val stream: DataStream<Pair<String, Long>> = ...  // Kotlin Pair → Kryo
```

Fix: используйте Flink `org.apache.flink.api.java.tuple.Tuple2`, не language-native tuples.

## Fail-fast: `pipeline.generic-types: false`

Single config setting, превращающий silent fallback в loud failure:

```yaml
# config.yaml
pipeline:
  generic-types: false
```

В Java code:

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.getConfig().disableGenericTypes();   // эквивалент
```

После этого любая попытка использовать `GenericTypeInfo<T>` в pipeline вызывает:

```
java.lang.IllegalArgumentException: 
Generic types have been disabled in the ExecutionConfig and type 
com.example.SomeClass is treated as a generic type.
```

Job **не стартует**. Видно сразу. Видно в CI integration tests. Видно локально при `mvn test`.

### Behavior после disableGenericTypes()

| Type | Без config | С `generic-types: false` |
|---|---|---|
| Flink POJO | OK (PojoTypeInfo) | OK |
| Tuple с `.returns()` | OK | OK |
| Tuple без `.returns()` (lambda) | InvalidTypesException | InvalidTypesException |
| Lombok без `@NoArgsConstructor` | Silent Kryo | **IllegalArgumentException** |
| POJO с `Instant` field | Silent Kryo для Instant | **IllegalArgumentException** |
| Avro Specific | OK (AvroTypeInfo) | OK |
| Protobuf (flink-protobuf) | OK (PbTypeInfo) | OK |
| Lambda через `Types.GENERIC(MyClass.class)` | OK Kryo | **IllegalArgumentException** (explicit Generic) |

Каждый break — early signal что нужно fix type info.

## Production migration story

Реальный кейс из production (anonymized):

**Симптом**: Fraud detection pipeline на 100k events/sec/subtask начал показывать backpressure после добавления нового event type `FraudAlert` с Lombok `@Data @Builder`. Throughput упал с 90M events/sec total на 12 TM до 55M (~40% degradation).

**Investigation**:

1. JobManager UI → Tasks → видим high backpressure на ProcessFunction после нового MapFunction.
2. TaskManager logs grep `"Falling back"`: десятки WARN записей `class FraudAlert ... falling back to GenericType`.
3. `javap -p FraudAlert.class` — подтверждаем что нет no-arg constructor (только `<init>(long, String, Instant)` от `@Builder`).

**Fix**:

```diff
-@Data @Builder
+@Data @Builder @NoArgsConstructor @AllArgsConstructor
 public class FraudAlert { ... }
```

Также:

```diff
 public class FraudAlert {
   private long alertId;
   private String userId;
-  private Instant detectedAt;        // ← Kryo для этого field
+  private long detectedAtMs;         // ← BasicTypeInfo
 }
```

После redeploy throughput вернулся к 90M.

**Prevention going forward**:

```yaml
# config.yaml
pipeline:
  generic-types: false
```

Plus integration test:

```java
@Test
public void noKryoFallbackInPipeline() {
    StreamExecutionEnvironment env = StreamExecutionEnvironment.createLocalEnvironment();
    env.getConfig().disableGenericTypes();   // explicit для теста
    buildPipeline(env);                       // та же логика что в production main
    env.getStreamGraph();                     // forces TypeInformation resolution
    // Если есть Kryo fallback, getStreamGraph() throws
}
```

После этих изменений за 6 месяцев zero Kryo regressions в production.

## Registering Kryo serializers (когда Kryo неизбежен)

Очень редко, но иногда class из third-party library не POJO-compatible и нет Avro alternative. Например, geo types from Esri, ML model types from Mahout. В таких случаях register custom Kryo serializer для **better Kryo performance**:

```java
import com.esotericsoftware.kryo.Serializer;

env.getConfig().registerTypeWithKryoSerializer(
    GeoPolygon.class,
    GeoPolygonKryoSerializer.class);

env.getConfig().registerKryoType(GeoPolygon.class);  // short ID, не FQN
```

```java
public class GeoPolygonKryoSerializer extends Serializer<GeoPolygon> {
    @Override
    public void write(Kryo kryo, Output output, GeoPolygon polygon) {
        output.writeInt(polygon.numVertices());
        for (Point p : polygon.vertices()) {
            output.writeDouble(p.lat());
            output.writeDouble(p.lng());
        }
    }

    @Override
    public GeoPolygon read(Kryo kryo, Input input, Class<? extends GeoPolygon> type) {
        int n = input.readInt();
        Point[] pts = new Point[n];
        for (int i = 0; i < n; i++) {
            pts[i] = new Point(input.readDouble(), input.readDouble());
        }
        return new GeoPolygon(pts);
    }
}
```

Registered Kryo с custom serializer достигает ~70% baseline (vs ~50% для unregistered). Это compromise, но **только когда альтернативы (POJO conversion, Avro wrapper) impossible**.

Также — registered Kryo с short class ID вместо FQN экономит ~30 bytes/record на wire.

## State considerations

Kryo serializer **не имеет schema evolution** в смысле Avro/POJO. State, written с Kryo, на upgrade может быть unreadable:

- Class FQN changed (rename, package move) → state unreadable, fail на recovery.
- New field added → Kryo пытается прочитать, видит EOF, fail или corrupt.
- Removed field → Kryo читает bytes, не находит куда положить, fail.

В contrast, POJO `PojoSerializer` имеет `PojoSerializerSnapshot` с reconciliation logic — добавление nullable field handled gracefully. Avro имеет full schema evolution. Kryo — ничего.

> **State + Kryo = upgrade trap.** При первой попытке upgrade Flink или refactor classes, ваш savepoint становится unreadable.

## Performance comparison

10M records single-thread bench:

```
Type                        Throughput      Bytes/record    CPU per record
─────────────────────────────────────────────────────────────────────────
POJO Flink                  28M rec/sec    ~24 bytes       0.7 μs
Avro Specific               23M rec/sec    ~22 bytes       0.9 μs
Tuple2                      30M rec/sec    ~16 bytes       0.65 μs
Protobuf native             22M rec/sec    ~20 bytes       0.95 μs
Kryo registered             14M rec/sec    ~30 bytes       1.5 μs
Kryo unregistered           10M rec/sec    ~60 bytes       2.0 μs
JSON (Jackson)              12M rec/sec    ~80 bytes       1.7 μs
```

Kryo unregistered — **3x slower** чем POJO, **6x larger** wire footprint. Production cost: 3x более dorogие TaskManager pods, 3x bigger network bandwidth, 3x slower state recovery.

## Production gotchas

- **`pipeline.generic-types: false` ломает существующие pipelines** — если вы deploy-ите этот config на legacy job, он failed на startup. Audit + fix всех types ПЕРЕД flipping.
- **WARN logs about fallback теряются в noise** — Flink логирует на INFO/WARN level, заглушаемые typical log filters. Без monitoring это invisible.
- **Lambda capture закрывает Kryo класс** — если lambda capture-ит non-Serializable instance, Flink сериализует closure через Kryo. Решение: capture только simple values, или make field static.
- **`ClosureCleaner` ошибки** — Flink имеет ClosureCleaner который nullifies unnecessary outer references. Если ClosureCleaner не справляется — instance передаётся через Kryo. См. WARN logs.
- **State Processor API не работает на Kryo state** — миграция Kryo → POJO для state requires manual ETL: write бывший Kryo state в external, восстановить как POJO.
- **`KryoSerializer` versioning** — Kryo lib upgrades между Flink versions могут break compatibility. Production rule: pin Kryo version и test savepoint compatibility before Flink upgrade.
- **Lombok `@With`** — генерирует withers (creates new instance с changed field), не нарушает POJO requirements, но не помогает с no-arg ctor.
- **Jackson `@JsonIgnore` fields** — не игнорируются Flink. Используйте `transient` modifier если хотите skip.
- **Concurrent classloading bugs** — Kryo внутри hot path может trigger classloader contention в high-parallelism deployments. Native POJO/Avro избегают этого.
- **JEP 411 (Security Manager) deprecation** — некоторые Kryo features (ReflectionFactory) могут не работать на JDK 17+ без `--add-opens`. Native serializers без проблем.

<Callout type="warning">
**Production checklist для preventing Kryo regression**:

1. `pipeline.generic-types: false` в production config.yaml.
2. `env.getConfig().disableGenericTypes()` в каждом Java main() для defense-in-depth.
3. Integration test `noKryoFallbackInPipeline()` который calls `env.getStreamGraph()`.
4. CI lint: search `.map(...)` без следующего `.returns(...)` — heuristic, ловит lambdas без type hints.
5. Code review checklist: новые types annotation-документировано как POJO.
6. Monitoring: alert на WARN log `Falling back to GenericType` (но это backstop, должно быть невозможно при config).

Это "defense in depth" — multiple layers ensure нет Kryo silently вкрадётся.
</Callout>

<Callout type="tip">
Если ваш existing codebase глубоко polluted Kryo, миграция incremental: (1) добавить `disableGenericTypes()` в новые job-ы, (2) для old job-ов добавить allow-list через `pipeline.allowed-generic-types: [GeoPolygon, LegacyType]` — fail для всех новых, allow listed. Постепенно сокращайте list. (3) Final step: remove allow-list, full ban.
</Callout>
