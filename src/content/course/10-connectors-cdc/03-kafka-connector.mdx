# Kafka connector вАФ primary Flink connector

Kafka вАФ —Б–∞–Љ—Л–є —З–∞—Б—В—Л–є source/sink –≤ Flink production. Flink Kafka connector implements FLIP-27 (Source) –Є FLIP-191 SinkV2 вАФ —Н—В–Њ modern, unified, supports exactly-once. –°—В–∞—А—Л–µ `FlinkKafkaConsumer`/`FlinkKafkaProducer` removed –≤ 2.0.

–≠—В–Њ—В —Г—А–Њ–Ї —А–∞–Ј–±–Є—А–∞–µ—В: KafkaSource —Б dynamic partition discovery, OffsetsInitializer variants, deserialization, KafkaSink —Б transactional EOS, configuration tuning, production gotchas.

## KafkaSource вАФ FLIP-27 implementation

```java
KafkaSource<Order> source = KafkaSource.<Order>builder()
    .setBootstrapServers("kafka-broker1:9092,kafka-broker2:9092")
    .setTopics("orders", "premium-orders")             // –Є–ї–Є setTopicPattern(Pattern.compile("orders-.*"))
    .setGroupId("order-processor")
    .setStartingOffsets(OffsetsInitializer.earliest())
    .setValueOnlyDeserializer(new OrderDeserializer())
    .setProperty("partition.discovery.interval.ms", "30000")  // dynamic discovery
    .build();

DataStream<Order> stream = env.fromSource(
    source,
    WatermarkStrategy.<Order>forBoundedOutOfOrderness(Duration.ofSeconds(10))
        .withTimestampAssigner((o, ts) -> o.getOrderTime())
        .withIdleness(Duration.ofMinutes(1)),
    "kafka-orders"
);
```

### Architecture

- **SplitEnumerator** = `KafkaSourceEnumerator` runs –љ–∞ JM. Discovers partitions —З–µ—А–µ–Ј `AdminClient.listTopics() + describeTopics()`. Periodic discovery –Ї–∞–ґ–і—Л–µ `partition.discovery.interval.ms`.
- **Split** = `KafkaPartitionSplit` вАФ `(topic, partition, startingOffset, stoppingOffset)`.
- **SourceReader** = `KafkaSourceReader` per subtask. Owns assigned splits, uses `KafkaConsumer` –і–ї—П fetching.

### Dynamic partition discovery

–≠—В–Њ **major improvement** over legacy FlinkKafkaConsumer:

```
T=0:   discover [orders-0, orders-1, ..., orders-7] вЖТ assign –Ї 8 subtasks
T=30s: discover [orders-0, ..., orders-7, orders-8, orders-9] вЖТ assign new partitions
T=60s: ...
```

–Ъ–Њ–≥–і–∞ Kafka admin adds partitions (–Є–ї–Є –љ–Њ–≤—Л–є topic matches pattern), Flink picks them up –±–µ–Ј restart. Property `partition.discovery.interval.ms` controls interval (default disabled вАФ `-1`, must enable explicitly).

**Production:** always enable –і–Є—Б–Ї–∞–≤–µ—А–Є –і–ї—П long-running jobs:

```java
.setProperty("partition.discovery.interval.ms", "30000")
```

## OffsetsInitializer variants

–Ъ–Њ–љ—В—А–Њ–ї—М –Њ—В–Ї—Г–і–∞ –љ–∞—З–Є–љ–∞—В—М reading:

```java
// Earliest available вАФ full backfill
.setStartingOffsets(OffsetsInitializer.earliest())

// Latest вАФ —В–Њ–ї—М–Ї–Њ new events
.setStartingOffsets(OffsetsInitializer.latest())

// From specific timestamp (event time)
.setStartingOffsets(OffsetsInitializer.timestamp(1735689600000L))

// From committed offsets, fallback earliest if no commit
.setStartingOffsets(OffsetsInitializer.committedOffsets(OffsetResetStrategy.EARLIEST))

// Custom (–љ–∞–њ—А–Є–Љ–µ—А per-partition)
.setStartingOffsets(OffsetsInitializer.offsets(Map.of(
    new TopicPartition("orders", 0), 100L,
    new TopicPartition("orders", 1), 200L
)))
```

### Stopping offsets вАФ –і–ї—П BOUNDED

```java
.setBounded(OffsetsInitializer.latest())         // stop at current end вЖТ BOUNDED source
.setBounded(OffsetsInitializer.timestamp(...))   // stop at timestamp
```

–° `setBounded()` source —Б—В–∞–љ–Њ–≤–Є—В—Б—П BOUNDED вЖТ Flink treats job –Ї–∞–Ї batch (–Ї–Њ–≥–і–∞ runtime mode = BATCH). Used –і–ї—П backfill jobs.

```java
// Unbounded (streaming):
.setStartingOffsets(...)                         // вЖР no setBounded

// Bounded (batch backfill):
.setStartingOffsets(...)
.setBounded(OffsetsInitializer.latest())
```

## Deserialization

Three styles:

### 1. Value-only

```java
.setValueOnlyDeserializer(new SimpleStringSchema())
.setValueOnlyDeserializer(new JsonDeserializationSchema<>(Order.class))
.setValueOnlyDeserializer(new AvroDeserializationSchema<>(Order.SCHEMA))
```

–Х—Б–ї–Є –љ—Г–ґ–љ–Њ —В–Њ–ї—М–Ї–Њ value вАФ most common.

### 2. With key, headers, metadata

```java
.setDeserializer(KafkaRecordDeserializationSchema.of(
    new KafkaDeserializationSchemaWrapper(...)
))
```

–Я–Њ–ї–љ—Л–є access –Ї ConsumerRecord (key, value, headers, topic, partition, offset, timestamp).

### 3. Confluent Schema Registry

```java
.setValueOnlyDeserializer(ConfluentRegistryAvroDeserializationSchema.forSpecific(
    Order.class, "http://schema-registry:8081"
))
```

Avro schema fetched dynamically, supports schema evolution.

## KafkaSink вАФ FLIP-191 SinkV2 implementation

```java
KafkaSink<Alert> sink = KafkaSink.<Alert>builder()
    .setBootstrapServers("kafka-broker1:9092")
    .setRecordSerializer(KafkaRecordSerializationSchema.builder()
        .setTopic("alerts")
        .setValueSerializationSchema(new AlertSerializationSchema())
        .build())
    .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
    .setTransactionalIdPrefix("fraud-detector-tx")
    .setProperty("transaction.timeout.ms", "900000")
    .build();

stream.sinkTo(sink);
```

### DeliveryGuarantee semantics

| Mode | Behaviour | Use case |
|---|---|---|
| `NONE` | Fire-and-forget вАФ –Љ–Њ–ґ–µ—В lose events | Metrics, non-critical |
| `AT_LEAST_ONCE` | Producer flushes on checkpoint, duplicates –љ–∞ failure | Idempotent consumer downstream |
| `EXACTLY_ONCE` | Two-Phase Commit —З–µ—А–µ–Ј Kafka transactions | Strict EOS |

### transactionalIdPrefix вАФ critical for recovery

```java
.setTransactionalIdPrefix("fraud-detector-tx")
```

Flink uses pattern `{prefix}-{subtask}-{checkpointId}` –і–ї—П transactional.id. –Э–∞ recovery Flink lists transactions matching prefix, aborts orphaned, resumes valid. **Prefix must be unique per job** вАФ –µ—Б–ї–Є –і–≤–µ jobs use same prefix, –Њ–љ–Є interfere вЖТ data loss / corruption.

<Callout type="warning">
Kafka `transactional.id` PITFALL: –і–Њ–ї–ґ–µ–љ –±—Л—В—М **—Г–љ–Є–Ї–∞–ї—М–љ—Л–є per job**. Blue/green deployment —Б –Њ–і–Є–љ–∞–Ї–Њ–≤—Л–Љ prefix = zombie fencing вАФ Kafka aborts ¬Ђ—Б—В–∞—А–Њ–≥–Њ¬ї producer-–∞, –љ–Њ–≤—Л–є job –њ–Њ–ї—Г—З–∞–µ—В aborted transactions, possible data loss. Encode environment –≤ prefix: `prod-fraud-v2-tx`, `staging-fraud-v2-tx`. –Я—А–Є major redeploy bump version suffix.
</Callout>

## SinkWriter, Committer, GlobalCommitter –≤ SinkV2

FLIP-191 SinkV2 introduces three components:

```java
public interface Sink<InputT> extends Serializable {
    SinkWriter<InputT> createWriter(InitContext context);
}

// Stateful variant —Б pre-commit:
public interface StatefulSink<InputT, WriterStateT> extends Sink<InputT> {
    StatefulSinkWriter<InputT, WriterStateT> createWriter(InitContext context);
    StatefulSinkWriter<InputT, WriterStateT> restoreWriter(InitContext context, Collection<WriterStateT> state);
    SimpleVersionedSerializer<WriterStateT> getWriterStateSerializer();
}

// Two-phase commit variant:
public interface TwoPhaseCommittingSink<InputT, CommT> extends Sink<InputT> {
    PrecommittingSinkWriter<InputT, CommT> createWriter(InitContext context);
    Committer<CommT> createCommitter() throws IOException;
    SimpleVersionedSerializer<CommT> getCommittableSerializer();
}
```

–Ф–ї—П exactly-once —Б Kafka: `TwoPhaseCommittingSink` вАФ pre-commit produces `Committable` (handle –Ї transaction), Committer calls `commitTransaction`.

–≠—В–Њ **strict improvement** over legacy `TwoPhaseCommitSinkFunction` вАФ clearer interfaces, decoupled writer –Њ—В committer, easier to test.

## –°—А–∞–≤–љ–Є—В–µ–ї—М–љ–∞—П —В–∞–±–ї–Є—Ж–∞: AT_LEAST_ONCE vs EXACTLY_ONCE

| Aspect | AT_LEAST_ONCE | EXACTLY_ONCE |
|---|---|---|
| Mechanism | Flush –љ–∞ checkpoint | Two-Phase Commit Kafka Tx |
| Latency overhead | Minimal | +1 round-trip –Ї Kafka coordinator |
| Throughput | ~95% of NONE | ~70-90% of NONE |
| Recovery semantics | Possible duplicates | –Э–Є–Ї–∞–Ї–Є—Е duplicates |
| Kafka requirement | Idempotent producer | Transactional + isolation.level=read_committed |
| Configuration complexity | Low | Higher (transactional.id prefix, timeouts) |
| Use case | Idempotent consumers | Strict accounting / financial |

## Production gotchas

### 1. `transaction.timeout.ms` < checkpoint interval вЖТ data loss

–Х—Б–ї–Є checkpoint takes longer than transaction timeout вАФ Kafka aborts transaction, but Flink –і—Г–Љ–∞–µ—В it's still alive вЖТ records lost.

```java
// Checkpoint interval = 10 min
env.enableCheckpointing(600_000);

// вЭМ Default transaction.timeout.ms = 60s вАФ too short!
// вЬЕ Set explicitly:
.setProperty("transaction.timeout.ms", "900000")    // 15 min вАФ safe margin
```

Plus Kafka broker has `transaction.max.timeout.ms` (default 15min) вАФ client cannot exceed this.

### 2. `transactionalIdPrefix` shared across deployments вЖТ corruption

–Х—Б–ї–Є –і–≤–µ Flink jobs (–Є–ї–Є —Б—В–∞—А–∞—П+–љ–Њ–≤–∞—П deployment) use same prefix:
- Kafka transactional.id —Г–љ–Є–Ї–∞–ї—М–љ—Л–є per producer instance
- Two producers —Б same transactional.id вЖТ Kafka aborts other (zombie fencing)
- Records lost / inconsistent state

**Fix:** unique prefix per job, encode environment (`prod-fraud-tx`, `staging-fraud-tx`).

### 3. Dynamic partition discovery disabled by default

```java
// вЭМ –Э–µ –њ–Њ–і—Е–≤–∞—В–Є—В –љ–Њ–≤—Л–µ partitions:
KafkaSource.<T>builder()
    .setTopics("orders")
    .build()
// 
// вЬЕ Discovery enabled:
    .setProperty("partition.discovery.interval.ms", "30000")
```

Legacy FlinkKafkaConsumer had it enabled by default. FLIP-27 KafkaSource вАФ disabled. Easy –Љ–Є–≥—А–∞—Ж–Є—П bug.

### 4. Group ID reused across Flink jobs

–Х—Б–ї–Є –љ–µ—Б–Ї–Њ–ї—М–Ї–Њ Flink jobs use same `setGroupId("processor")`:
- Kafka thinks –Њ–љ–Є –≤ –Њ–і–љ–Њ–є consumer group
- Re-assigns partitions –Љ–µ–ґ–і—Г –љ–Є–Љ–Є
- Both jobs –љ–µ –њ–Њ–ї—Г—З–∞—О—В –њ–Њ–ї–љ—Л–є –і–∞–љ–љ—Л–µ

–Ґ–∞–Ї–ґ–µ Flink NOT –Є—Б–њ–Њ–ї—М–Ј—Г–µ—В Kafka committed offsets –і–ї—П recovery вАФ –Њ–љ uses Flink checkpoint state. Group ID matters —В–Њ–ї—М–Ї–Њ –і–ї—П observability (lag monitoring) and committed offsets visibility.

**Practice:** unique group ID per Flink job, even –≤ case-—П—Е —А–µ—Б—В–∞—А—В–∞ (—Е–Њ—В—П same job restart should reuse).

### 5. Watermark idleness –і–ї—П idle partitions

```java
WatermarkStrategy.<Order>forBoundedOutOfOrderness(Duration.ofSeconds(10))
    .withIdleness(Duration.ofMinutes(1))    // вЖР critical
```

–Х—Б–ї–Є –Њ–і–Є–љ partition idle (no events) вАФ –±–µ–Ј idleness watermark stuck –≤ last seen ts вЖТ downstream operators (CEP, windows) wait forever.

### 6. Schema evolution —Б Avro вАФ careful

–Я—А–Є rolling schema evolution –≤ Confluent registry:
- Producers write –љ–Њ–≤–Њ–є schema
- Flink consumers —Б —Б—В–∞—А–Њ–є schema decoder вЖТ –Љ–Њ–≥—Г—В fail
- Use forward/backward compatible changes (add optional field, –љ–µ remove)

```java
.setValueOnlyDeserializer(ConfluentRegistryAvroDeserializationSchema.forSpecific(
    Order.class, "http://schema-registry:8081", schemaRegistryConfig))
```

Test schema evolution –љ–∞ staging –њ–µ—А–µ–і production deploy.

## Key takeaways

- **KafkaSource** вАФ FLIP-27 implementation, **KafkaSink** вАФ FLIP-191 SinkV2.
- **Dynamic partition discovery** —З–µ—А–µ–Ј `partition.discovery.interval.ms` вАФ must enable explicitly.
- **OffsetsInitializer** controls start (earliest/latest/timestamp/committed/custom) –Є stop (BOUNDED via setBounded).
- **EXACTLY_ONCE** —З–µ—А–µ–Ј Two-Phase Commit Kafka transactions вАФ `transactionalIdPrefix` must be unique per job, `transaction.timeout.ms` > checkpoint interval.
- **SinkV2 TwoPhaseCommittingSink** clearly separates writer (pre-commit) –Є committer (commit) вАФ strict improvement over legacy.
- **withIdleness** –Њ–±—П–Ј–∞—В–µ–ї–µ–љ –і–ї—П production —Б multiple partitions / sources.
