# Capstone — observability: OpenTelemetry, Grafana, alerting

Phase 5: production observability. Без proper observability streaming pipeline — black box. Когда что-то ломается (а ломается часто на scale), нужна visibility в каждый слой: source lag, network buffers, state size, checkpoint duration, sink throughput, downstream impact.

Capstone использует **OpenTelemetry** для metrics + traces (Flink 2.x has native OTel exporter), **Prometheus** для metrics storage, **Grafana** для visualization, **AlertManager** для routing.

## Alerting rules

```yaml
groups:
- name: capstone-streaming
  interval: 30s
  rules:

  # CRITICAL — page on-call
  - alert: PipelineJobDown
    expr: up{job="flink-capstone"} == 0
    for: 1m
    labels:
      severity: critical
      team: streaming
    annotations:
      summary: "Capstone pipeline JobManager down"
      runbook: "https://wiki/runbooks/capstone-jm-down"

  - alert: CheckpointFailureRate
    expr: increase(flink_jobmanager_job_numFailedCheckpoints[10m]) > 3
    for: 5m
    labels: { severity: critical }
    annotations:
      summary: "Multiple checkpoint failures detected"

  - alert: WatermarkSeverelyLagging
    expr: |
      max(time() - flink_taskmanager_job_task_operator_currentInputWatermark{operator="UserAggregator"} / 1000) > 300
    for: 5m
    labels: { severity: critical }
    annotations:
      summary: "Watermark lag > 5 minutes — data freshness impact"

  - alert: PostgresReplicationSlotLag
    expr: pg_replication_slot_lag_bytes > 50e9   # 50 GB
    for: 5m
    labels: { severity: critical }
    annotations:
      summary: "Postgres replication slot > 50 GB lag — risk of disk fill"

  # WARNING — Slack channel
  - alert: BackpressureHigh
    expr: |
      avg(flink_taskmanager_job_task_operator_backPressuredTimeMsPerSecond) > 700
    for: 10m
    labels: { severity: warning }
    annotations:
      summary: "Sustained backpressure > 70% — possible bottleneck"

  - alert: LateEventsHigh
    expr: |
      rate(flink_taskmanager_job_task_operator_numLateRecordsDropped[5m]) > 100
    for: 5m
    labels: { severity: warning }
    annotations:
      summary: "Late events rate elevated — investigate source lag"

  - alert: StateSizeGrowth
    expr: |
      predict_linear(flink_taskmanager_job_task_operator_state_size_in_bytes[1h], 86400 * 7) > 1e12
    for: 1h
    labels: { severity: warning }
    annotations:
      summary: "State projected to exceed 1TB в 7 дней — TTL not effective?"

  - alert: SinkCommitSlow
    expr: paimon_snapshot_commit_duration_seconds{quantile="0.99"} > 60
    for: 15m
    labels: { severity: warning }
    annotations:
      summary: "Paimon commit P99 > 60s — compaction tuning needed"

  # INFO — dashboard only
  - alert: AutoscalerScaleUp
    expr: changes(flink_jobmanager_taskmanager_count[30m]) > 2
    labels: { severity: info }
    annotations:
      summary: "Multiple autoscaler events — workload changed"
```

  group_by: ['alertname', 'severity']

## Distributed tracing

Flink 2.x has experimental traces support. Each record can carry trace context, propagated через operators.

```java
// В custom operator emit trace span
public void processElement(EnrichedOrder order, Context ctx, Collector<UserEvent> out) {
    Span span = openTelemetry.getTracer("capstone")
        .spanBuilder("UserAggregator.processElement")
        .setAttribute("user_id", order.userId())
        .setAttribute("order_id", order.orderId())
        .startSpan();
    try (Scope scope = span.makeCurrent()) {
        // ... business logic
        out.collect(...);
    } finally {
        span.end();
    }
}
```

Use cases:
- Trace single event через pipeline (source → aggregator → sink) для debugging.
- Identify slow stages — span duration breakdown.
- Correlate logs (через trace_id) with metrics for postmortems.

Sampling: 1% по default (cost). 100% sampling для error spans.

## Runbooks

Critical: каждый alert должен иметь linked runbook. Without runbook — on-call helpless.

```markdown
# Runbook: WatermarkSeverelyLagging

## Symptoms
Alert fires when watermark lag > 5 minutes for sustained 5 min period.
Real impact: window emits delayed, downstream metrics stale, billing/anomaly detection compromised.

## Diagnostic steps
1. Open Grafana "Capstone End-to-End" dashboard.
2. Identify which operator has lag — check "Watermark Lag" panel by operator.
3. Check "Backpressure heatmap" — is there bottleneck causing slow processing?
4. Check Postgres replication slot lag — is source behind?
5. Check Kafka consumer lag (if applicable).
6. Check Flink job manager logs for restart events.

## Likely causes & fixes
- **Source lag**: Check Postgres replication slot lag. If > 5GB, may indicate Flink CDC was paused. Restart pipeline if needed.
- **Bottleneck on UserAggregator**: scale up parallelism. `kubectl patch flinkdeployment ... '{"spec":{"job":{"parallelism": <N+8>}}}'`. Autoscaler should pick up too.
- **Bottleneck on Paimon sink**: Check compaction lag. Manual compact if behind.
- **Bottleneck on RocksDB compaction**: Tune `state.backend.rocksdb.predefined-options: FLASH_SSD_OPTIMIZED`.

## Escalation
If not resolved in 30 minutes, escalate to streaming-team Slack.
```

## Production gotchas

- **Cardinality explosion в Prometheus labels** — `user_id` as label = millions of series. Strip high-cardinality labels at collector level.
- **OTel collector OOM** — if Flink emits too many metrics, collector overload. Configure batching, sampling.
- **Tempo storage** — traces volume огромный. Tail-based sampling — keep only errors + slow.
- **Loki query cost** — full-text search expensive. Index by labels, use structured logs для queries.
- **Alert fatigue** — too many warnings → ignored. Tune thresholds based на actual SLO violations, not arbitrary.
- **Dashboard sprawl** — каждый team создаёт свои dashboards, hard to find canonical. Maintain one «golden» dashboard per service.
- **Runbook drift** — runbooks становятся stale. Periodic review (quarterly) обязателен.
- **Metric retention** — high-resolution (10s) для 30 days, downsampled (5min) для year. Cost-effective.
- **AlertManager grouping** — без grouping каждый instance gets alert → notification storm. `group_by: [alertname]` minimum.

<Callout type="warning">
**Observability — не «set and forget»**. Dashboards stale, alerts no longer relevant, runbooks outdated — это normal entropy. Schedule quarterly observability review: are dashboards used? Are alerts firing correctly (false positives, false negatives)? Are runbooks accurate? Treat observability как ongoing investment, не one-time setup.
</Callout>
