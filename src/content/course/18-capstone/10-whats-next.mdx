# Capstone — что дальше: career path, conferences, ecosystem

Курс закончен. Вы прошли путь от «что такое stream processing» до production-grade capstone pipeline с CDC, stateful processing, EOS, K8s autoscaling, chaos testing, observability. Это substantive скилл — не «знание API», а способность дизайнить и operate real-time data systems в production.

Этот урок — карта дальнейшего изучения. Куда идти, что читать, с кем общаться, и как deepen skills в направлениях которые матёр в 2026+.

---

## Где вы сейчас

После курса вы способны:

- **Design** stateful streaming architecture для real production use case (fraud detection, real-time analytics, CDC pipelines, recommendations).
- **Implement** Flink job с proper state management, watermark strategy, EOS sinks.
- **Deploy** на Kubernetes с autoscaling, HA, blue/green upgrades.
- **Operate** — chaos testing, observability, on-call response.
- **Debug** typical production issues — backpressure, checkpoint failures, state growth.

Это middle-senior level Flink expertise. Senior+ — это depth (deep RocksDB internals, custom operators, SQL planner contributions) и breadth (broader ecosystem, multiple streaming engines, data architecture).

## Advanced Flink topics

### Apache Paimon deep

Capstone использовал Paimon basics. Advanced topics:

- **Append-only tables** для streaming OLAP без UPSERT overhead.
- **Bucket mode = -1** — dynamic bucketing, automatic re-balancing.
- **Compact procedures** — scheduled compaction strategies для large tables.
- **Branches и tags** — Git-like versioning over data, A/B testing on data.
- **Materialized views in Paimon** — declarative streaming pipelines.

References:
- `paimon.apache.org` docs.
- Source code: `github.com/apache/paimon`.

### RisingWave / Materialize alternatives

Modern streaming SQL DBs — different paradigm от Flink (specialized для SQL aggregations vs general).

- **Materialize** — Differential Dataflow based. Incremental view maintenance. Sweet spot: real-time dashboards с zero staleness.
- **RisingWave** — distributed streaming SQL, PostgreSQL compatible. Sweet spot: streaming SQL aggregations.

Why learn:
- Comparison perspective — when Flink overkill, when these tools fit.
- Career — these tools growing market, related role pipeline.
- Architecture decisions — knowing options means making informed tradeoffs.

References:
- Materialize blog — high quality streaming SQL content.
- RisingWave docs — strong tutorials для streaming SQL basics.

### Flink Agents — emerging area

Flink Agents preview discussed в M17/09. Watch this space:

- FLIP-531 — actively evolving, breaking changes expected through 1.0.
- Community Slack `#flink-agents` channel.
- Early adopter case studies — likely papers / blog posts from Alibaba, Netflix через 2026-2027.

### Custom operators / extensions

Beyond using Flink — extending it:

- **Custom Source connectors** — implement FLIP-27 Source API для proprietary sources.
- **Custom Sink** — SinkV2 API для new sinks.
- **Table API UDFs** — scalar, table, async functions.
- **Custom State backend** — extremely advanced, rare need.

Path: read existing connectors (`flink-connector-kafka`, `flink-connector-jdbc`) as reference; small custom extension; eventual upstream contribution.

### Internals deep dive

Если ROAD к Flink committer / deep expertise:

- **RocksDB / ForSt internals** — LSM tree, compaction strategies, JNI overhead.
- **Network stack** — Netty layer, credit-based protocol implementation.
- **Memory management** — managed memory allocation, off-heap buffers.
- **Scheduler** — Adaptive Scheduler internals, in-place rescaling mechanics.
- **Checkpoint coordination** — ABS algorithm, barrier alignment, async snapshots.

References:
- Flink Improvement Proposals (FLIPs) — `cwiki.apache.org/confluence/display/FLINK/Flink+Improvement+Proposals`.
- VLDB papers — Flink contributions (Apache Flink: Unified Stream and Batch Processing — original; Disaggregated State paper — 2025).
- Source code with strategic depth (start from `StreamExecutionEnvironment.execute()` → JobGraph → ExecutionGraph → Scheduler).

## Broader ecosystem

### Adjacent technologies

- **Apache Kafka** — Flink complementary. Deep Kafka knowledge invaluable (partitioning, transactions, KIP-848 cooperative rebalancing).
- **Apache Iceberg** — alternative к Paimon для lakehouse. Stronger ecosystem (Snowflake, BigQuery, Trino support).
- **Apache Arrow** — columnar in-memory format. Increasingly important in cross-system zero-copy.
- **DuckDB** — embedded analytics — useful для testing, validation.
- **dbt** — analytics engineering — complementary к streaming для downstream transforms.
- **Trino / Presto** — interactive SQL over lake — common pairing with Flink pipelines.

### MLOps integration

Modern streaming + ML — growing intersection:

- **Feature stores** — Feast, Tecton, Hopsworks. Integration patterns с Flink.
- **Model serving** — TensorFlow Serving, ONNX Runtime, BentoML. ML_PREDICT vs external.
- **MLflow / Weights & Biases** — model versioning, experiments tracking.
- **Vector databases** — Pinecone, Qdrant, Weaviate. VECTOR_SEARCH в Flink 2.2.

### Cloud-specific knowledge

Cloud providers offer managed Flink:

- **AWS** — Managed Service for Apache Flink (formerly Kinesis Data Analytics).
- **Confluent** — Flink in Confluent Cloud (kafka + flink integrated).
- **Aiven** — Managed Flink + Kafka.
- **Ververica Platform** — commercial Flink platform.

Managed services tradeoffs: less ops burden, less flexibility, vendor lock-in.

## Career paths

### Senior Streaming Engineer

Focus: deeply understand Flink + ecosystem, lead production deployments, mentor others. Path: 3-5 years с Flink production exposure, contributions back (bug fixes, features), conference talks.

### Streaming Architect

Focus: design data architectures using streaming, evaluate technologies, set patterns. Path: senior engineer + cross-team architecture experience + strong business communication.

### Platform Engineer (Streaming Platform)

Focus: building internal Flink-as-a-Platform — multi-tenant clusters, self-service onboarding, observability. Path: K8s + Flink + DevOps experience combined.

### Flink Committer / Apache Member

Focus: open source contribution. Path: meaningful contributions, mailing list participation, FLIP authorship, eventual nomination.

### Data Engineer (Streaming Specialty)

Focus: end-to-end data pipelines using Flink + Kafka + Iceberg + dbt. Path: data engineering background + streaming depth.

### ML Platform Engineer (Streaming + AI)

Emerging area. Focus: real-time ML inference, feature stores, agent platforms. Combines Flink + ML/AI knowledge.

## Open source contribution

Path к committer / community member:

1. **Use Flink at scale** — accumulate production experience.
2. **Report bugs constructively** — minimal reproductions, clear context.
3. **Fix small bugs** — start with «good first issue» tagged items.
4. **Add tests / documentation** — accepted contributions, less risk.
5. **Propose feature / write FLIP** — only after demonstrating community presence.
6. **Review others' PRs** — important for committer pipeline.
7. **Speak at meetups / conferences** — visibility.

Realistic timeline: 1-2 years sustained contribution для committer recognition.

## Continuous learning

Streaming evolving rapidly. Strategies для staying current:

- **Subscribe Apache Flink release notes** — major releases ~6 months.
- **Follow Flink Slack** — first-line awareness of new features.
- **Quarterly review** — what's new in Flink, Paimon, Iceberg, Kafka. Adjust skills accordingly.
- **Practical experimentation** — when new feature released, try it in side project. Hands-on > reading.
- **Network with other Flink engineers** — meetups, Slack DMs, conference networking.
- **Internal knowledge sharing** — present к team, write internal docs. Best way to solidify knowledge.
