# Capstone — source layer: Flink CDC из Postgres

Первая phase implementation: получить changes из Postgres в Paimon raw tables. Это foundation — все последующие layers reading from raw tables.

Mы используем **Flink CDC 3.6 directly** (без intermediate Kafka), как обосновано в design phase. Pipeline declarative YAML, под капотом Flink DataStream graph с EOS sink.

---

## Step 1: Postgres preparation

```sql
-- На primary Postgres (postgresql.conf):
ALTER SYSTEM SET wal_level = 'logical';
ALTER SYSTEM SET max_wal_senders = 20;
ALTER SYSTEM SET max_replication_slots = 20;
ALTER SYSTEM SET max_slot_wal_keep_size = '100GB';  -- safety bound

-- Restart Postgres

-- Создать user для replication:
CREATE USER flink_cdc WITH REPLICATION LOGIN PASSWORD 'secret';
GRANT CONNECT ON DATABASE ecommerce TO flink_cdc;
GRANT USAGE ON SCHEMA public TO flink_cdc;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO flink_cdc;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO flink_cdc;

-- Подготовить tables для CDC:
ALTER TABLE customers REPLICA IDENTITY DEFAULT;  -- uses PK
ALTER TABLE orders    REPLICA IDENTITY FULL;     -- full row для UPDATE/DELETE before-values
ALTER TABLE order_items REPLICA IDENTITY FULL;
ALTER TABLE products  REPLICA IDENTITY DEFAULT;

-- Publication:
CREATE PUBLICATION flink_cdc_pub FOR TABLE
    customers, orders, order_items, products;

-- Replication slot:
SELECT pg_create_logical_replication_slot('flink_cdc', 'pgoutput');
```

REPLICA IDENTITY FULL для orders/order_items — потому что нам нужны old values при UPDATE/DELETE для downstream change events (например, для transition tracking status).

## Step 2: Flink CDC YAML pipeline

```yaml
source:
  type: postgres
  hostname: postgres-replica.internal  # ← read replica, не primary!
  port: 5432
  username: flink_cdc
  password: secret
  database-name: ecommerce
  schema-name: public
  table-name: public\\..*                # все public tables
  slot.name: flink_cdc
  decoding.plugin.name: pgoutput
  scan.startup.mode: initial
  scan.incremental.snapshot.enabled: true
  scan.incremental.snapshot.chunk.size: 100000
  chunk.meta.group.size: 1000
  connect.max-retries: 5
  jdbc.properties.tcpKeepAlive: true

sink:
  type: paimon
  catalog.properties.metastore: hive
  catalog.properties.uri: thrift://hive-metastore:9083
  catalog.properties.warehouse: s3://capstone/paimon
  table.properties.bucket: 8
  table.properties.changelog-producer: input
  table.properties.merge-engine: deduplicate
  table.properties.snapshot.expire.execution-mode: async
  table.properties.snapshot.num-retained.min: 30
  table.properties.snapshot.num-retained.max: 1000

transform:
  - source-table: public.orders
    projection: |
      *,
      amount * 0.01 AS amount_dollars,
      DATE_FORMAT(created_at, 'yyyy-MM-dd') AS order_date,
      CASE WHEN amount > 100000 THEN 'large'
           WHEN amount > 10000 THEN 'medium'
           ELSE 'small' END AS order_size,
      NOW() AS ingested_at
    description: Orders с derived columns

  - source-table: public.customers
    projection: |
      *,
      UPPER(email) AS email_normalized,
      DATE_PART('year', AGE(NOW(), created_at)) AS account_age_years
    filter: deleted_at IS NULL
    description: Active customers с derived fields

  - source-table: public.order_items
    projection: '*'
    description: Order items raw

  - source-table: public.products
    projection: |
      *,
      CASE WHEN inventory_count = 0 THEN 'out_of_stock'
           WHEN inventory_count < 10 THEN 'low_stock'
           ELSE 'available' END AS stock_status
    description: Products с derived

route:
  - source-table: public.customers
    sink-table: capstone.dim_customers
  - source-table: public.orders
    sink-table: capstone.fact_orders
  - source-table: public.order_items
    sink-table: capstone.fact_order_items
  - source-table: public.products
    sink-table: capstone.dim_products

pipeline:
  name: capstone-postgres-cdc
  parallelism: 8
  schema.change.behavior: try-evolve   # production prod safety
  execution.runtime-mode: streaming
  execution.checkpointing.interval: 60s
  execution.checkpointing.mode: EXACTLY_ONCE
  state.backend.type: forst
  state.backend.forst.primary-dir: s3://capstone/cdc-state
```

Critical choices in this config:

1. **Read replica для source** — `postgres-replica.internal`. Production primary остаётся без CDC load.
2. **Incremental snapshot** — lock-free, parallelizable. Чанки 100k rows.
3. **schema.change.behavior: try-evolve** — apply when possible, log warning otherwise. Production prod safety против destructive changes.
4. **transform expressions** — derived columns computed during ingest. Downstream queries simpler.
5. **filter** для customers — пропускаем deleted_at IS NOT NULL records (soft-deleted users not in lake).
6. **Paimon bucket=8** — matches Flink parallelism. Bigger bucket = more parallelism in compaction, but more small files.

## Step 4: Initial snapshot strategy

На production tables с 100M+ rows initial snapshot — самая чувствительная phase. Если ошибиться:
- Snapshot загружает Postgres → latency приложения spike.
- Snapshot занимает дни.
- Disk фnam ouverloaded.

Strategy для capstone:

1. **Запуск snapshot ночью** — minimum traffic window. 0:00-6:00 UTC.
2. **Read replica с minimum lag** — verify pg_replication_slot lag < 1min перед запуском.
3. **Parallelism начально 4** — не 8 во время snapshot. После snapshot complete — scale up через autoscaler.
4. **Chunk size 100k** — balance между request size и memory footprint.
5. **Monitoring** — `SELECT * FROM pg_stat_activity WHERE query LIKE 'COPY%'` watch postgres-side load.

```sql
-- Pre-snapshot validation:
SELECT slot_name, restart_lsn,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS lag
FROM pg_replication_slots WHERE slot_name = 'flink_cdc';
```

## Step 5: Verification после snapshot

```sql
-- В Paimon
SELECT
    table_name,
    snapshot_id,
    schema_id,
    commit_user,
    commit_time,
    record_count
FROM capstone.`fact_orders$snapshots`
ORDER BY snapshot_id DESC
LIMIT 10;

-- Compare row counts с Postgres
SELECT COUNT(*) FROM capstone.fact_orders;
-- vs
SELECT COUNT(*) FROM postgres.public.orders;
```

Counts должны совпадать с tolerance ~1min lag (events во время snapshot).

## Step 6: Schema evolution test

Verify schema evolution works (controlled experiment в staging):

```sql
-- В Postgres staging:
ALTER TABLE orders ADD COLUMN discount_code VARCHAR(50);

-- В Postgres:
INSERT INTO orders (..., discount_code) VALUES (..., 'SUMMER2026');

-- В Paimon через 1-2 минуты:
SELECT * FROM capstone.fact_orders WHERE discount_code IS NOT NULL LIMIT 10;
-- Должны видеть new column populated
```

Если column добавлен auto — schema evolution propagation работает.

Для production prod: добавьте explicit step в release process — manual review schema changes перед apply.

- name: flink-cdc
  rules:
    - alert: ReplicationSlotLag
      expr: pg_replication_slot_lag_bytes > 50000000000  # 50 GB
      for: 5m

        severity: critical

        flink_jobmanager_job_uptime > 7200  # 2h
        and
        flink_taskmanager_job_task_operator_snapshot_phase_active == 1
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: Snapshot phase took > 2 hours, investigate

      expr: flink_taskmanager_job_task_operator_currentInputWatermark < (time() - 60) * 1000
      for: 5m

        severity: warning

## Migration plan от existing infrastructure

If team has existing Debezium → Kafka → Flink:

1. Deploy Flink CDC parallel pipeline в staging.
2. Compare row counts + sample data between two pipelines for accuracy.
3. Switch downstream consumers (Paimon queries) на new pipeline atomically.
4. Decommission old Debezium connectors.

Не «migrate in place» — too risky. Parallel + cutover.

## Production gotchas (Phase 1)

- **Snapshot loads Postgres replica** — verify replica can handle. Sometimes replica behind primary by-design (read-only replica for analytics) — snapshot pressure compounds. Mitigation: increase replica resources during snapshot window.
- **Network failures during snapshot** — Flink CDC retries, но some implementations требуют restart полного snapshot. Set `connect.max-retries: 5+`, monitor recovery.
- **Replication slot orphaning** — если Flink CDC job stopped permanently, slot остаётся, WAL накапливается. Cleanup procedure: `SELECT pg_drop_replication_slot('flink_cdc');` после confirming pipeline retired.
- **Time zones и timestamps** — Postgres TIMESTAMP без TZ vs WITH TZ — proper handling в transform. Default Flink CDC uses UTC; ensure downstream consistent.
- **Numeric precision loss** — DECIMAL(20,10) в Postgres → handle через `decimal.handling.mode=precise` (string-encoded) для finance. Default `double` имеет precision loss.
- **TOAST values** — large text/jsonb > 8KB. По умолчанию Debezium шлёт unchanged placeholder в UPDATE. Flink CDC handles это, но downstream might see placeholder values. Solution: explicit unavailable.value.placeholder handling.
- **Schema registry vs no registry** — Flink CDC может работать без Schema Registry для simplicity, но в production add Confluent SR или Apache Atlas для schema governance.

<Callout type="warning">
**REPLICA IDENTITY FULL имеет performance impact на Postgres writes** — каждый UPDATE/DELETE on table требует logging full old row в WAL. Это increased WAL volume + network traffic. На critical OLTP tables измеряйте impact в staging. Для some tables может быть acceptable DEFAULT (только PK) если downstream не нуждается в before-values.
</Callout>
