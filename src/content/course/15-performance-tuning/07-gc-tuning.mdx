# GC tuning для streaming

GC pause — главная причина latency spikes в Flink job-ах. Stateful streaming на heap (HashMapStateBackend), event objects, watermark generators — все это создаёт GC pressure. На high-throughput pipeline с 100MB/sec потоком объектов GC может занимать 5-15% CPU времени, а отдельные pauses — десятки или сотни миллисекунд.

В Flink 2.x (Java 17/21 required) выбор GC принципиально шире чем раньше: G1 (default), ZGC (generational, GA в Java 21), Shenandoah (production-ready). В этом уроке препарируем когда какой выбрать и как тюнить.

---

## GC landscape Java 17/21 для streaming

| GC | Pause | Throughput | Heap range | Использовать когда |
|---|---|---|---|---|
| **G1** (default) | 10-200ms | High | 4GB-32GB | General-purpose, default outdoor pipelines |
| **ZGC** (gen, Java 21+) | < 1ms | Medium-High | 8GB-16TB | Low-latency требование, large heap |
| **Shenandoah** (gen) | < 10ms | Medium | 4GB-1TB | Альтернатива ZGC, более mature for some workloads |
| Parallel GC | 100-1000ms | Highest | Any | Batch (НЕ для streaming) |
| Serial GC | High | Low | < 100MB | Никогда для production |

Java 17 LTS — minimum для Flink 2.x. Java 21 LTS добавил **generational ZGC** — это game changer для low-latency.

## G1 (Garbage-First) — default

G1 разбивает heap на **regions** (~2-32MB), young/old generations динамические. Concurrent mark, incremental copy.

```
[Heap]
[Y][Y][O][O][Y][H][O][O][Y][O][...]
 Y = young region
 O = old region
 H = humongous (>50% region size)
```

**Pros**:
- Pause time goal (`MaxGCPauseMillis`) — soft target.
- Concurrent mark not stop-the-world.
- Stable performance до 32GB heap.

**Cons**:
- Pauses 10-200ms typical, не sub-millisecond.
- На very large heap (>32GB) degrades.
- Humongous objects (большие arrays) ломают region allocator.

### G1 tuning для Flink

```yaml
env.java.opts.taskmanager: >
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=100
  -XX:G1HeapRegionSize=4m
  -XX:InitiatingHeapOccupancyPercent=35
  -XX:+ParallelRefProcEnabled
  -XX:G1ReservePercent=20
  -XX:+UseStringDeduplication
```

Параметры:

- `MaxGCPauseMillis=100` — soft goal 100ms pause. Lower = более частые pauses, выше CPU.
- `G1HeapRegionSize=4m` — explicit region size (avoid auto-calculation на edge sizes).
- `InitiatingHeapOccupancyPercent=35` — начать concurrent mark при 35% old gen (default 45%). Раннее старт = меньше шансов на full GC.
- `G1ReservePercent=20` — reserve 20% heap для evacuation failures. Защита от full GC под burst.
- `UseStringDeduplication` — JVM ищет identical String objects и shares — экономит heap на high-cardinality string keys.

## ZGC — low pause champion

ZGC использует **colored pointers** и **load barriers** — concurrent everything, кроме trivial root scan. Java 21 LTS — generational ZGC GA (раньше был только flat).

**Pros**:
- Pauses **< 1ms** даже на TB heap.
- Generational в Java 21 — efficiency сравнима с G1 для большинства workloads.
- Не degrades с увеличением heap.

**Cons**:
- 5-15% throughput cost.
- Memory overhead ~3% (colored pointer metadata).
- На small heap (<8GB) overhead не worth.

### ZGC tuning для Flink (Java 21+)

```yaml
env.java.opts.taskmanager: >
  -XX:+UseZGC
  -XX:+ZGenerational
  -XX:MaxGCPauseMillis=10
  -XX:+AlwaysPreTouch
  -XX:-OmitStackTraceInFastThrow
  -Xms8g
  -Xmx8g
```

Параметры:

- `+UseZGC` + `+ZGenerational` — generational ZGC (Java 21).
- `MaxGCPauseMillis=10` — target pause (soft, обычно достигается).
- `+AlwaysPreTouch` — touch all heap pages at startup. Предотвращает page faults during runtime (commit RSS upfront).
- `-Xms == -Xmx` — fixed heap size (ZGC любит stable heap).

**Best practice**: Java 21 LTS + generational ZGC — это recommended для latency-critical Flink jobs.

## Shenandoah — альтернатива ZGC

Shenandoah — concurrent GC от Red Hat, included в OpenJDK 17+. Pauses sub-10ms, similar idea к ZGC (concurrent compaction).

**Когда выбирать вместо ZGC**:
- Java 17 (где generational ZGC не доступен — только Java 21+).
- Workload-specific bench показывает better behavior.

```yaml
env.java.opts.taskmanager: >
  -XX:+UseShenandoahGC
  -XX:+ShenandoahUncommit
  -XX:ShenandoahUncommitDelay=300000
  -XX:+AlwaysPreTouch
```

В большинстве cases **ZGC predominates** для streaming. Shenandoah — backup plan.

## GC выбор: decision tree

```
Heap size?
├─ < 4GB
│   └─ G1 (overhead ZGC/Shenandoah не worth)
│
├─ 4-32GB
│   ├─ Latency-critical (P99 < 50ms)?
│   │   ├─ Java 21+ → ZGC generational
│   │   └─ Java 17 → Shenandoah
│   └─ General-purpose → G1 (default)
│
└─ > 32GB
    └─ ZGC (Java 21+) или Shenandoah (Java 17)
```

```promql
# G1 metrics:

flink_taskmanager_Status_JVM_Memory_Heap_Used
```

Healthy indicators:

| Metric | Healthy | Action |
|---|---|---|
| GC time per minute (G1 Young) | < 5000ms (8%) | > 10000ms → switch к ZGC, или increase heap |
| GC time per minute (ZGC) | < 1000ms (1.5%) | > 3000ms → check allocation rate, larger heap |
| Heap_Used / Heap_Max | < 75% peak | > 90% sustained → bigger heap, или leak |
| GC count rate | Stable | Spikes coincide с throughput peaks → tune |

## JFR (Java Flight Recorder) для diagnose

JFR — low-overhead profiler. Continuous recording можно держать в production:

```yaml
env.java.opts.taskmanager: >
  -XX:+FlightRecorder
  -XX:StartFlightRecording=duration=0,filename=/var/log/flink-jfr.jfr,settings=profile
  -XX:FlightRecorderOptions=stackdepth=64,maxsize=100M
```

JFR collects: GC events, allocation stack traces, thread states, contention. Analyze с JDK Mission Control:

- GC pause duration histogram.
- Allocation rate per class.
- Where time spent (CPU profile).

Идеально для production troubleshooting — overhead < 1%.

## Diagnostic flow для GC issues

Симптом: периодические latency spikes, throughput dips.

```
1. Метрики GC time per minute:
   - G1 Old > 5000ms/min → full GC events → critical
   - Sum (Young + Old) > 10s/min → tune

2. JFR recording 5-10 minutes:
   - Pause duration histogram
   - Allocation profile (which class allocates most)

3. Анализ allocation hotspots:
   - Avro/Protobuf objects? → optimize codec
   - String objects? → enable +UseStringDeduplication
   - byte[] buffers? → check ByteBuffer reuse

4. Heap headroom:
   - Heap used > 80% sustained → bigger heap
   - Heap fluctuates 30%-90% → garbage spikes from operator

5. Switch GC если G1 не tunable:
   - ZGC (Java 21) для < 10ms pauses
```

## Production tuning patterns

### Pattern 1: Stateless transform (light heap)

```yaml
taskmanager.memory.process.size: 4g

env.java.opts.taskmanager: >
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=100
  -XX:+UseStringDeduplication
```

G1 — best balance для small heap.

### Pattern 2: Stateful с HashMapStateBackend (heap-heavy)

```yaml
taskmanager.memory.process.size: 16g
taskmanager.memory.managed.fraction: 0.1   # state на heap, не managed

env.java.opts.taskmanager: >
  -XX:+UseZGC
  -XX:+ZGenerational
  -XX:MaxGCPauseMillis=10
  -XX:+AlwaysPreTouch
```

State on heap → большая нагрузка GC. ZGC mandatory.

### Pattern 3: RocksDB state + lightweight processing

```yaml
taskmanager.memory.process.size: 16g
taskmanager.memory.managed.fraction: 0.5   # большой RocksDB

env.java.opts.taskmanager: >
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=100
  -XX:G1HeapRegionSize=4m
  -XX:InitiatingHeapOccupancyPercent=35
```

State в RocksDB (managed memory, off-heap), heap небольшой → G1 OK.

### Pattern 4: Low-latency real-time (sub-50ms P99)

```yaml
taskmanager.memory.process.size: 16g

env.java.opts.taskmanager: >
  -XX:+UseZGC
  -XX:+ZGenerational
  -XX:MaxGCPauseMillis=5
  -XX:+AlwaysPreTouch
  -XX:-OmitStackTraceInFastThrow
  -Xms14g
  -Xmx14g
```

Aggressive ZGC config для absolute minimum pause.

### Pattern 5: Large state HashMap backend (legacy, не recommended)

```yaml
taskmanager.memory.process.size: 32g

env.java.opts.taskmanager: >
  -XX:+UseZGC
  -XX:+ZGenerational
  -XX:MaxGCPauseMillis=10
  -XX:+AlwaysPreTouch
  -Xms28g
  -Xmx28g
```

Generational ZGC — единственный viable выбор для 32GB heap streaming.

## Allocation reduction strategies

GC tuning только лечит симптомы — root cause часто allocation rate. Reduction strategies:

### 1. Reuse objects в hot path

```java
public class ReuseMapper extends RichMapFunction<Event, Result> {
    private transient Result reusable;

    @Override
    public void open(Configuration cfg) {
        reusable = new Result();
    }

    @Override
    public Result map(Event e) {
        reusable.userId = e.getUserId();
        reusable.count = e.getCount();
        return reusable;  // ← reuse instance
    }
}
```

Caveat: только если downstream не holdит reference. Не работает если operator chained → next operator reads sync OK. Не работает если sink буферизует references.

### 2. Avoid String concatenation в hot path

```java
// BAD: allocates new String каждую запись
String key = e.getUserId() + "_" + e.getCategory();

// GOOD: pre-computed compose
String key = ConcatPool.intern(e.getUserId(), e.getCategory());
```

Или explicit `StringBuilder` reuse.

### 3. Primitive collections (Eclipse Collections, fastutil)

```java
// BAD: 24 bytes per Long в HashMap<Long,Long>
HashMap<Long, Long> map = new HashMap<>();

// GOOD: int-keyed primitive map
Long2LongHashMap map = new Long2LongHashMap();   // 16 bytes total per entry
```

Critical для large in-memory state.

### 4. Pooled ByteBuffer для serialization

```java
private static final BufferPool POOL = ...;

byte[] serialize(Event e) {
    ByteBuffer buf = POOL.acquire();
    try {
        // serialize into buf
        return Arrays.copyOf(buf.array(), buf.position());
    } finally {
        POOL.release(buf);
    }
}
```

## Production gotchas

### Gotcha 1: Set Xmx без AlwaysPreTouch

```yaml
env.java.opts: "-Xmx16g"
```

JVM аллоцирует virtual memory upfront, но physical pages только по requested. На burst (e.g., GC) — page faults → latency spikes 10-100ms.

**Fix**: `-XX:+AlwaysPreTouch` — touch all pages на старте, повышает startup time но eliminates runtime spikes.

### Gotcha 2: Xms != Xmx

```yaml
-Xms4g -Xmx16g
```

JVM может shrink heap во время idle, потом aggressively grow обратно → GC pressure spikes.

**Fix**: `-Xms == -Xmx`. Fixed heap, predictable.

### Gotcha 3: ZGC + small heap = overhead

ZGC overhead ~3% memory + ~10% throughput. На 2GB heap это значимая трата.

**Fix**: ZGC только при heap >= 8GB.

### Gotcha 4: G1 humongous objects

Allocations > 50% of G1 region size = humongous. Stored separately, не cleaned до full GC. Big Avro/Protobuf objects могут triggerить.

**Diagnose**: GC log shows `G1 Humongous Allocation`. JFR показывает large allocations.

**Fix**: `-XX:G1HeapRegionSize=8m` (bigger regions), или reduce object size.

### Gotcha 5: -XX:+OmitStackTraceInFastThrow

Default JVM omits stack traces for re-thrown exceptions для performance. Но в production debugging критично иметь полные traces.

**Fix**: `-XX:-OmitStackTraceInFastThrow` — full traces. Minor performance cost.

### Gotcha 6: GC log в K8s ephemeral storage

```yaml
-Xlog:gc*:file=/tmp/gc.log:time,uptime
```

В Kubernetes /tmp может быть tmpfs (memory) → memory pressure. Использовать persistent volume или disable file logging.

### Gotcha 7: Java version drift

Flink 2.x build на Java 17 — но runtime на Java 21. Generational ZGC доступен только runtime Java 21+. Verify:

```bash
docker exec <tm-pod> java -version
```

Если Java 17 runtime + `-XX:+ZGenerational` → JVM startup fails.

<Callout type="warning">
GC tuning — это последняя линия optimization, не первая. Если у вас GC pressure — сначала проверьте: (1) allocation hotspots в JFR; (2) leaks в HashMapStateBackend; (3) heap headroom (Heap_Used / Heap_Max > 80%? bigger heap). Только потом tune GC algorithm. Иначе вы маскируете симптомы, не решая root cause.
</Callout>
