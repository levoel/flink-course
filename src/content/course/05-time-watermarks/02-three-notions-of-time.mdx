# Три понятия времени в Flink

Время — самая коварная абстракция в stream processing. В отличие от batch (где время записи известно из источника), streaming работает с непрерывным потоком, в котором события могут приходить **не в том порядке**, в котором они произошли. Flink выделяет три разных понятия времени, и выбор между ними определяет, будет ли ваш job **детерминированным** или нет.

Этот урок объясняет, почему почти все production Flink jobs должны использовать **event time**, и какие подводные камни ждут тех, кто выбирает processing time «потому что проще».

### 1. Event time — когда событие произошло

**Embedded в record**. Назначается источником (mobile app пишет `timestamp` поле в JSON, IoT sensor — UNIX epoch, web server — request received time).

```java
public class Transaction {
    long userId;
    long eventTime;   // ← это event time, заполнен producer-ом
    BigDecimal amount;
}
```

Flink извлекает event time через `TimestampAssigner`:

```java
WatermarkStrategy
    .<Transaction>forBoundedOutOfOrderness(Duration.ofSeconds(20))
    .withTimestampAssigner((tx, recordTs) -> tx.getEventTime());
```

**Свойства event time**:
- **Детерминированный** — при повторной обработке (replay из Kafka offset) результат идентичен
- **Out-of-order** — события могут приходить в произвольном порядке из-за network jitter, retry, parallelism
- **Требует watermarks** — для определения, когда «достаточно» данных собрано для emit окна

### 2. Processing time — wall clock TaskManager-а

`System.currentTimeMillis()` на момент обработки записи. **Самый быстрый** режим — никакой логики watermarks, окна просто закрываются по таймеру.

```java
stream
    .keyBy(Transaction::getUserId)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
    .process(...);
```

**Свойства processing time**:
- **Низкая latency** — нет ожидания watermark
- **Простая семантика** — окно закрывается ровно в указанный момент
- **НЕдетерминированный** — replay из того же offset даёт **другие** результаты
- **Зависит от инфраструктуры** — GC pause 200мс может сдвинуть события между окнами

### 3. Ingestion time — момент входа в Flink

Время, когда запись пересекла границу Flink Source. Источник timestamp — TaskManager wall clock в момент `SourceReader.pollNext()`.

**В Flink 2.x этот режим removed как отдельная TimeCharacteristic**. Можно эмулировать через custom TimestampAssigner:

```java
.withTimestampAssigner((event, recordTs) -> System.currentTimeMillis())
```

Но это almost never правильное решение — теряет преимущества event time без выигрыша processing time.

### Концептуальная разница

| Сценарий | Event time | Processing time |
|---|---|---|
| Live processing | Окно `[10:00, 10:05)` закрывается когда watermark > 10:05 | Окно закрывается когда wall clock > 10:05 |
| Replay из 7-дневной давности | Окно `[10:00, 10:05)` 7 дней назад закрывается корректно | Все записи попадают в текущее окно — broken |
| Backfill из S3 + live Kafka | Hybrid source, events merge по timestamp | Bound to ingestion order — broken |
| Late event (timestamp 9:58, пришло в 10:07) | Попадает в `[9:55, 10:00)` или late side output | Попадает в `[10:05, 10:10)` — wrong window |
| GC pause 5 sec на operator | Окно эмитит ровно в правильное время | Окно может пропустить 5 sec данных |

### Реальный кейс: миграция fraud detector

Команда мигрирует с processing time на event time. На replay 1 дня данных через replay из Kafka:

```java
// PROCESSING TIME — broken on replay
.window(TumblingProcessingTimeWindows.of(Time.minutes(5)))

// EVENT TIME — replay correct, but требует watermark + TimestampAssigner
.assignTimestampsAndWatermarks(
    WatermarkStrategy.<Transaction>forBoundedOutOfOrderness(Duration.ofSeconds(30))
        .withTimestampAssigner((tx, ts) -> tx.getEventTime())
        .withIdleness(Duration.ofMinutes(1))
)
.window(TumblingEventTimeWindows.of(Time.minutes(5)))
```

При reprocessing с event time job катится через Kafka offset с **maximum throughput** (no rate limiting), но aggregations остаются bit-exact. Это и есть **lambda architecture без lambda** — один pipeline для batch и stream.

## Когда processing time всё-таки уместен

Есть редкие, но валидные use case для processing time:

### 1. Service-level monitoring

Метрика: «сколько API requests в последнюю минуту». Здесь wall clock — правильное определение «минуты»:

```java
stream
    .window(TumblingProcessingTimeWindows.of(Time.minutes(1)))
    .aggregate(new CountAgg());
```

### 2. Rate limiting / throttling

Логика «emit не чаще, чем раз в N секунд» естественно processing time:

```java
.process(new KeyedProcessFunction<>() {
    public void processElement(Event e, Context ctx, Collector out) {
        long now = ctx.timerService().currentProcessingTime();
        if (now - lastEmitTime.value() > 1000) {
            out.collect(e);
            lastEmitTime.update(now);
        }
    }
});
```

### 3. Side effects с wall clock семантикой

«Очистить state если ничего не приходило 1 час» — wall clock 1 час, не event time:

```java
ctx.timerService().registerProcessingTimeTimer(now + Time.hours(1).toMilliseconds());
```

### Правило большого пальца

| Use case | Time semantic |
|---|---|
| Бизнес-агрегация (sum, count, avg на временном окне) | **Event time** |
| Stateful detection (fraud, anomaly) | **Event time** |
| Join двух потоков по времени | **Event time** |
| Rate metrics (req/sec) | Processing time |
| TTL очистка state | Processing time |
| Reactive alerts (notify if N events in M minutes) | Зависит: bizness — event time, infra — processing time |

## Hybrid: timestamps for assignment, processing time for triggers

Flink позволяет смешивать: event time для windowing, processing time для emission через **custom triggers**:

```java
stream
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .trigger(
        ContinuousProcessingTimeTrigger.of(Time.seconds(30))   // emit каждые 30s
    )
    .aggregate(new SumAgg());
```

Это полезно для **early firing** — показывать промежуточный результат раньше, чем watermark достигнет границы окна.

## Multi-source: разные источники, разный event time

Когда job читает несколько Kafka topics, каждый со своим event time, Flink propagates **MIN(input watermarks)** через operator. Это означает:

- Slow source держит watermark всего pipeline
- Idle source держит watermark в прошлом
- Backlogged source (replay) может «убежать вперёд» по event time

Решения:
- `withIdleness(Duration)` — пометить source как idle при отсутствии данных
- **Watermark Alignment (FLIP-217)** — паузить fast sources (детально в уроке 05)

Эти проблемы и подходы — основная тема следующих 3 уроков.

## Production gotchas

### 1. Mixed time semantics в одной job

Использовать event time для windowing, processing time для cleanup timer — OK. Но смешивать event time tumbling window с processing time tumbling window в одной job — путь к hard-to-reason bugs. Старайтесь делать time semantics однородным per logical pipeline.

### 2. Wall clock skew между TaskManager-ами

Processing time зависит от wall clock каждого TaskManager-а. NTP drift в датацентре может быть 100ms+. Если разные TaskManager-ы видят немного разный wall clock — processing time окна закроются в разное время. Очень редкая, но воспроизводимая причина «нондетерминированных» багов. Event time не зависит от wall clock.

### 3. Timezone в event time

Event time = UNIX epoch millis (UTC). Если producer пишет timestamp в local timezone и забыл конвертировать — события «вернутся назад во времени» при DST transition. **Всегда UTC** в event time.

<Callout type="warning">
Event time всегда UTC epoch millis. Любой local timezone (особенно с DST) приводит к non-monotonic timestamps при переходе на летнее/зимнее время — watermark скачет назад, окна закрываются дважды или не закрываются. Producer должен конвертировать в UTC ДО отправки.
</Callout>

### 4. Reordering из retry

Producer retry: запись отправлена в 12:00:00, потерялась, retry в 12:00:01 успешен. На consumer side — event time 12:00:00, processing time 12:00:01. Если использовать processing time → событие приходит «позже» чем должно. Event time правильно сохранит исходное время.

### 5. Long event time gaps

Если события приходят раз в час (rare events), processing time windowing работает мгновенно (закрывается каждые 5 мин по wall clock). Event time будет ждать watermark — а watermark двигается только когда есть новые события. Без `withIdleness` окно может никогда не закрыться. Это не баг event time — это правильное поведение, но надо понимать.

<Callout type="tip">
Production default: event time для всей бизнес-логики (windowing, joins, fraud detection), processing time только для wall-clock метрик (req/sec, rate limiting) и TTL cleanup. Смешивание двух semantics в одной агрегации почти всегда баг.
</Callout>

### 6. Timestamp assigner возвращает null или старый timestamp

Если `TimestampAssigner` возвращает `Long.MIN_VALUE` или старый timestamp, **watermark не движется** — окна не закроются никогда. Type-safe защита:

```java
.withTimestampAssigner((event, recordTs) -> {
    if (event.getEventTime() == 0) {
        // лучше fallback на Kafka timestamp чем Long.MIN_VALUE
        return recordTs;
    }
    return event.getEventTime();
})
```

## Cheat sheet

```
Event time   = когда событие произошло (embedded in record)
             → детерминированный, требует watermarks
             → выбор по умолчанию

Processing   = System.currentTimeMillis() на TaskManager-е
             → быстрый, но недетерминированный
             → только для wall-clock метрик и rate limiting

Ingestion    = wall clock на момент входа в Flink
             → deprecated в 2.x, эмулируется через TimestampAssigner
             → almost never правильный выбор
```
