# Flink Metrics System

Метрики — главная observability primitive для Flink. Web UI красив для live debugging, но historical trends, alerting, capacity planning требуют persistent metrics. Flink имеет богатую metrics систему с 4 типами метрик, иерархической MetricGroup-аппой, и плагинов exporters (Prometheus, OTel, JMX, StatsD).

В этом уроке препарируем 4 типа метрик (Counter / Gauge / Meter / Histogram), MetricGroup hierarchy, и как добавить user metrics в свой код правильно.

---

## Четыре типа метрик

### 1. Counter — accumulated count

Monotonically increasing integer counter. Можно increment/decrement, но обычно только increment.

```java
public class MyMapper extends RichMapFunction<Event, Result> {
    private transient Counter errorCounter;
    private transient Counter processedCounter;

    @Override
    public void open(Configuration cfg) {
        errorCounter = getRuntimeContext()
            .getMetricGroup()
            .counter("errors");

        processedCounter = getRuntimeContext()
            .getMetricGroup()
            .counter("processedRecords");
    }

    @Override
    public Result map(Event e) {
        try {
            Result r = process(e);
            processedCounter.inc();
            return r;
        } catch (Exception ex) {
            errorCounter.inc();
            throw ex;
        }
    }
}
```

Counter подходит для:
- Records processed
- Errors / retries
- Cache hits / misses
- Custom business events

В Prometheus это **counter** type, в Grafana обычно visualized as rate: `rate(my_counter[5m])`.

### 2. Gauge — instantaneous value

Текущее значение, может go up or down. Lambda evaluated каждый sample.

```java
public class StatefulMapper extends RichMapFunction<Event, Result> {
    private transient Map<String, Long> userCache = new HashMap<>();

    @Override
    public void open(Configuration cfg) {
        getRuntimeContext().getMetricGroup()
            .gauge("cacheSize", () -> (long) userCache.size());

        getRuntimeContext().getMetricGroup()
            .gauge("memoryUsedMB", () ->
                Runtime.getRuntime().totalMemory() / 1024 / 1024);
    }

    @Override
    public Result map(Event e) {
        userCache.computeIfAbsent(e.getUserId(), k -> 0L);
        return process(e);
    }
}
```

Gauge подходит для:
- Queue depth
- Cache size
- Memory usage
- Connection pool active connections

В Prometheus это **gauge** type, в Grafana `my_gauge` напрямую.

**Performance**: lambda calculation вызывается на каждый scrape (e.g. 15s в Prometheus). Дорогие calculations → noticeable overhead. Cache при необходимости.

### 3. Meter — events per second

Rate measurement. Track records/sec, throughput.

```java
public class MyMapper extends RichMapFunction<Event, Result> {
    private transient Meter recordsPerSecond;

    @Override
    public void open(Configuration cfg) {
        com.codahale.metrics.Meter dropwizardMeter =
            new com.codahale.metrics.Meter();
        recordsPerSecond = getRuntimeContext()
            .getMetricGroup()
            .meter("recordsPerSecond",
                new DropwizardMeterWrapper(dropwizardMeter));
    }

    @Override
    public Result map(Event e) {
        recordsPerSecond.markEvent();   // increment event count
        return process(e);
    }
}
```

Meter computes:
- `getRate()` — events/sec
- `getCount()` — total events

Это convenience wrapper — можно тоже самое через Counter + Gauge для rate. Прометheus auto-computes rate из counter.

**В большинстве случаев** — используйте Counter, считайте rate в Prometheus query (`rate(counter[1m])`). Meter избыточен в этой setup.

### 4. Histogram — distribution

Distribution measurements: latency, sizes, etc. Compute percentiles (P50, P95, P99).

```java
public class AsyncLookup extends AsyncFunction<Event, Result> {
    private transient Histogram lookupLatency;

    @Override
    public void open(Configuration cfg) {
        com.codahale.metrics.Histogram dwHist =
            new com.codahale.metrics.Histogram(
                new SlidingTimeWindowReservoir(60, TimeUnit.SECONDS));
        lookupLatency = getRuntimeContext()
            .getMetricGroup()
            .histogram("dbLookupLatencyMs",
                new DropwizardHistogramWrapper(dwHist));
    }

    @Override
    public void asyncInvoke(Event input, ResultFuture<Result> resultFuture) {
        long start = System.currentTimeMillis();
        db.lookupAsync(input.getId(), (result, error) -> {
            long elapsed = System.currentTimeMillis() - start;
            lookupLatency.update(elapsed);
            resultFuture.complete(Collections.singleton(result));
        });
    }
}
```

Histogram queries:
- `getStatistics().getQuantile(0.99)` — P99
- `getStatistics().getMean()` — average
- `getCount()` — total samples

**Reservoir matters**: `SlidingTimeWindowReservoir(60s)` — rolling 60s window. Без этого histogram использует `ExponentiallyDecayingReservoir` — биased для recent samples но не sharp window.

В Prometheus histogram type экспортируется как multiple metrics: `_bucket`, `_count`, `_sum`.

## MetricGroup hierarchy

Метрики организованы в **MetricGroup** иерархию:

```
<host>.taskmanager.<tm_id>.<job_name>.<task_name>.<subtask_index>.<operator_name>.<metric_name>
```

Пример:

```
host1.taskmanager.tm-1.MyJob.KeyedProcess.0.HotKeyDetector.errors
host1.taskmanager.tm-1.MyJob.KeyedProcess.1.HotKeyDetector.errors
```

`getRuntimeContext().getMetricGroup()` возвращает operator-level group. Из этого можно создавать sub-groups:

```java
MetricGroup base = getRuntimeContext().getMetricGroup();

MetricGroup userGroup = base.addGroup("userMetrics");
MetricGroup categoryGroup = userGroup.addGroup("category", "fraud");
// → user_metrics.category.fraud.<metric>

Counter alerts = categoryGroup.counter("alerts");
```

Это даёт structured naming, легче alerts/dashboards filter by tag.

### Variable scopes

Default scope для operator:
```
taskmanager.<tm_id>.<job_name>.<task_name>.<subtask>.<operator_name>
```

Можно override через config:
```yaml
metrics.scope.tm: "<host>.taskmanager.<tm_id>"
metrics.scope.tm.job: "<host>.taskmanager.<tm_id>.<job_name>"
metrics.scope.task: "<host>.taskmanager.<tm_id>.<job_name>.<task_name>.<subtask_index>"
metrics.scope.operator: "<host>.taskmanager.<tm_id>.<job_name>.<task_name>.<subtask_index>.<operator_name>"
```

Эти templates контролируют labels в Prometheus exporter.

## Naming conventions

Flink не enforce conventions, но есть de-facto standards:

| Metric type | Naming pattern | Example |
|---|---|---|
| Counter | `<noun>` или `<noun>Total` | `processedRecords`, `errorsTotal` |
| Gauge | `<noun>` либо `<noun>Bytes` | `cacheSize`, `memoryUsedBytes` |
| Meter | `<noun>PerSecond` | `recordsPerSecond` |
| Histogram | `<noun>Ms` либо `<noun>Latency` | `dbLookupLatencyMs`, `serializeTimeNs` |

Bad naming:
- `latency` → не понятно units. Use `latencyMs`.
- `count` → counter or gauge? Use `totalCount` (counter) или `currentCount` (gauge).
- `time` → units? `timeMs`, `timeNs`.

### Tags vs metric names

Plain names:
```
errors.payment
errors.billing
errors.fraud
```

3 metrics. Cardinality высокая, графики разбросаны.

Tags approach:
```java
MetricGroup base = getRuntimeContext().getMetricGroup();
base.addGroup("category", category).counter("errors");
```

1 metric `errors` с label `category`. Меньше metrics names, queryable как `sum by (category)`. Prometheus loves это.

## Metric reporters

Reporters экспортируют metrics к external systems. Конфиг:

```yaml
metrics.reporters: prom

metrics.reporter.prom.factory.class: org.apache.flink.metrics.prometheus.PrometheusReporterFactory
metrics.reporter.prom.port: 9249
metrics.reporter.prom.filterLabelValueCharacters: true
```

Поддерживаемые reporters:

| Reporter | Use case |
|---|---|
| **PrometheusReporter** (pull) | Стандарт для metrics в Kubernetes |
| **PrometheusPushGatewayReporter** | Short-lived jobs (batch) |
| **OpenTelemetryReporter** (1.20+, 2.3 gRPC GA) | Unified observability stack |
| **InfluxDBReporter** | InfluxDB / TimescaleDB stacks |
| **GraphiteReporter** | Legacy Graphite stacks |
| **StatsDReporter** | Datadog (через statsd) |
| **SLF4JReporter** | Logging metrics (debug only) |
| **JMXReporter** | Local JMX inspection |

Можно include multiple reporters concurrently:

```yaml
metrics.reporters: prom,otel

metrics.reporter.prom.factory.class: org.apache.flink.metrics.prometheus.PrometheusReporterFactory
metrics.reporter.prom.port: 9249

metrics.reporter.otel.factory.class: org.apache.flink.metrics.otel.OtelReporterFactory
metrics.reporter.otel.endpoint: http://otel-collector:4317
```

## Production patterns

### Pattern 1: Business KPI metrics

```java
public class FraudDetector extends KeyedProcessFunction<String, Event, Alert> {
    private transient Counter fraudDetected;
    private transient Counter falsePositives;
    private transient Histogram detectionLatency;

    @Override
    public void open(Configuration cfg) {
        MetricGroup g = getRuntimeContext().getMetricGroup()
            .addGroup("business");
        fraudDetected = g.counter("fraudDetected");
        falsePositives = g.counter("falsePositives");
        detectionLatency = g.histogram("detectionLatencyMs",
            new DropwizardHistogramWrapper(
                new com.codahale.metrics.Histogram(
                    new SlidingTimeWindowReservoir(60, TimeUnit.SECONDS))));
    }

    @Override
    public void processElement(Event e, Context ctx, Collector<Alert> out) {
        long start = System.currentTimeMillis();
        boolean isFraud = check(e);
        detectionLatency.update(System.currentTimeMillis() - start);
        if (isFraud) {
            fraudDetected.inc();
            out.collect(new Alert(e));
        }
    }
}
```

Business metrics, alerting на fraud rate spike: `rate(business_fraudDetected[5m]) > 10`.

### Pattern 2: External system health

```java
public class DbLookup extends RichAsyncFunction<Event, Result> {
    private transient Counter dbCallsTotal;
    private transient Counter dbCallsFailures;
    private transient Histogram dbCallLatency;
    private transient Gauge<Long> dbConnPoolActive;

    @Override
    public void open(Configuration cfg) {
        MetricGroup g = getRuntimeContext().getMetricGroup().addGroup("db");
        dbCallsTotal = g.counter("callsTotal");
        dbCallsFailures = g.counter("callsFailures");
        dbCallLatency = g.histogram("callLatencyMs", ...);
        g.gauge("connectionsActive", () -> connectionPool.getActive());
    }
    // ...
}
```

Monitor external dependency health: error rate, latency, connection pool depth.

### Pattern 3: State-related debug metrics

```java
public class StatefulProcessor extends KeyedProcessFunction<...> {
    private transient ValueState<UserHistory> state;
    private transient Histogram stateSizeBytes;

    @Override
    public void open(Configuration cfg) {
        state = getRuntimeContext().getState(...);
        stateSizeBytes = getRuntimeContext().getMetricGroup()
            .histogram("stateSizeBytes", ...);
    }

    @Override
    public void processElement(...) {
        UserHistory h = state.value();
        stateSizeBytes.update(h == null ? 0 : h.estimateSize());
        // ...
    }
}
```

Track state size distribution per key — detect skew, abnormal users.

## Production gotchas

### Gotcha 1: Histogram reservoir matters

```java
new com.codahale.metrics.Histogram();  // default ExponentiallyDecayingReservoir
```

Default reservoir biased для recent samples — но это не sharp window. Для P99 за «last 1 minute» используйте `SlidingTimeWindowReservoir`.

```java
new com.codahale.metrics.Histogram(
    new SlidingTimeWindowReservoir(60, TimeUnit.SECONDS));
```

### Gotcha 2: Per-key metrics = cardinality explosion

```java
// ANTI-PATTERN:
getMetricGroup()
    .addGroup("user", record.getUserId())   // millions of users
    .counter("events");
```

Million-cardinality label → millions of metrics → Prometheus dies.

**Fix**: top-N tracking — track only top 10 hot users:

```java
private final Map<String, Counter> hotUserCounters = new HashMap<>();
// На каждый record: если userId в top-10 (через HyperLogLog или Count-Min), track separately.
```

### Gotcha 3: Gauge lambda дорогая

```java
g.gauge("slowMetric", () -> {
    return database.expensiveQuery();   // called every scrape!
});
```

Каждые 15s (Prometheus default) тяжёлая операция → blocks metric reporter.

**Fix**: cache value в RichFunction, update lazy:

```java
private volatile long cachedValue = 0;
@Override
public void open(Configuration cfg) {
    // Update periodically через timer
    new Thread(() -> {
        while (running) {
            cachedValue = database.expensiveQuery();
            Thread.sleep(60_000);
        }
    }).start();
    g.gauge("metric", () -> cachedValue);
}
```

### Gotcha 4: Counter overflow

`org.apache.flink.metrics.Counter` использует `long`, overflow невозможно практически. Но если используете `int` counter from Dropwizard через wrapper — overflow возможен.

### Gotcha 5: Metric name conflicts

Two operators named same → metrics merge with same name. Confusion в Grafana.

**Fix**: explicit operator names:

```java
.uid("my-mapper")
.name("MyMapper")
```

`name()` идёт в metric path → unique names = unique metrics.

### Gotcha 6: Reporter overhead

Prometheus reporter scraping → CPU/network spike каждые 15s. На huge cluster это noticeable.

**Fix**: filter unused metrics:

```yaml
metrics.reporter.prom.scope-filter: "!flink.taskmanager.*.shuffle.*"
```

Exclude expensive subsystems вроде network shuffle internals.

### Gotcha 7: Time units inconsistent

Built-in metrics use mix of nanoseconds, milliseconds, seconds. `currentInputWatermark` — Unix milliseconds. `checkpointStartDelayNanos` — nanoseconds. Verify units перед alerting!

## Diagnostic checklist для metrics

1. **Are metrics being exported?** Прometheus targets discoverable?
2. **Counter rate**: `rate(counter[5m])` non-zero?
3. **Histogram percentiles**: P50/P95/P99 reasonable?
4. **Gauge fluctuation**: stable plateau или линейный рост?
5. **Cardinality**: metric explosion?
6. **Naming consistency**: easy filter и query?

---

<Callout type="note">
Counter — самый используемый тип metrics. Если сомневаетесь — Counter. Rate computation делайте в Prometheus (`rate()`), не в Meter — это flexibility (window size variable, multi-rate views).
</Callout>

---

<Callout type="warning">
Per-key labels (userId, accountId, sessionId) — cardinality explosion. Prometheus может умереть с 1M+ unique series. Используйте top-N tracking или explicit aggregation. Лучше всего — track aggregate (counter + sum by category), не individual entities.
</Callout>
