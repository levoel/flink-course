# Runtime Data Sampling (FLIP-570)

В production streaming pipelines регулярно возникает проблема: «у нас один operator замедлен / выдаёт странные результаты, но мы не знаем что внутри потока данных». До FLIP-570 опции были:
1. Логировать каждую запись — flood logs, performance disaster.
2. Добавить debug код, redeploy job — слишком медленно для urgent prod issue.
3. Угадывать на основе aggregate metrics — slow root cause discovery.

**FLIP-570 Runtime Data Sampling** (Flink 2.3+) даёт: **live sampling записей** через Web UI без redeploy. Кликнули «Sample 100 records» на operator → видите реальные данные через 1-2 секунды. Это game changer для production debugging.

---

## Что такое Runtime Data Sampling

Web UI новый tab **«Sample»** на operator detail page. Пользователь:
1. Выбирает operator + subtask.
2. Выбирает `input` или `output` поток.
3. Sample size (e.g., 100 records).
4. Optional filter (predicate в SQL-like expression).
5. Click «Sample».

Через ~1-2 секунды отображается:

```
Sample of 100 records from Map.output (subtask 5):

#1   {userId: "u_481923", amount: 14.99, category: "books"}
#2   {userId: "u_481923", amount: 14.99, category: "books"}  ← DUPLICATE
#3   {userId: "u_481923", amount: 14.99, category: "books"}  ← DUPLICATE
#4   {userId: "u_481923", amount: 14.99, category: "books"}
...
```

Immediate insight: subtask 5 видит дубликаты для одного user. Probably broken deduplication upstream.

## Use cases

### Use case 1: Data skew investigation

Симптом: subtask 12 busy 100%, остальные idle. Подозрение skew.

Sample input на subtask 12 (100 records):

```
80 records: userId = "u_whale_001"
20 records: various
```

Confirmed: hot user (whale). Identified в 30 seconds, без adding metrics.

### Use case 2: Data quality issues

Симптом: downstream alerting на missing fields.

Sample output operator A (where transformation happens):

```
#1   {userId: "u_001", amount: 14.99, category: "books"}
#5   {userId: null, amount: 14.99, category: "books"}     ← MISSING userId
#23  {userId: "u_017", amount: -5.00, category: "books"}  ← NEGATIVE amount
#67  {userId: "u_044", amount: 14.99, category: ""}       ← EMPTY category
```

Identified: upstream sometimes produces null/negative/empty fields. Filter в parser нужен.

### Use case 3: Schema evolution issues

После schema change downstream consumer ломается. Sample output:

```
#1   {userId: "u_001", amount: 14.99, category: "books", newField: "ABC"}
#2   {userId: "u_002", amount: 14.99, category: "books", newField: "DEF"}
#3   {userId: "u_003", amount: 14.99, category: "books"}  ← MISSING newField
```

Identified: some records pre-schema-evolution. Need migration или handle gracefully.

### Use case 4: Deduplication verification

Sample with filter `userId = "u_001"` через 1 минуту:

```
Time 14:01:23: {userId: "u_001", txn_id: "abc"}
Time 14:01:23: {userId: "u_001", txn_id: "abc"}   ← DUPLICATE!
Time 14:01:24: {userId: "u_001", txn_id: "def"}
```

Confirmed: deduplication broken — same txn_id passes twice.

### Use case 5: Watermark / event-time issues

Sample output после window operator:

```
#1   window=[14:00, 14:05], userId="u_001", count=5
#2   window=[14:00, 14:05], userId="u_001", count=2   ← LATE!
```

Two emissions для same window — late events triggered allowedLateness. Investigate watermark generator.

## Configuration

```yaml
# Включить sampling (default enabled в 2.3+):
rest.flamegraph.refresh-interval: 60000

runtime-sampling.enabled: true
runtime-sampling.max-records-per-request: 1000
runtime-sampling.max-bytes-per-request: 10485760    # 10MB cap
runtime-sampling.timeout: 30s
runtime-sampling.allow-output-sampling: true
runtime-sampling.allow-input-sampling: true
```

### Security: production access control

```yaml
runtime-sampling.access-control.enabled: true
runtime-sampling.access-control.required-role: flink-debugger
```

Default — open access. Production должен иметь auth (через REST API security layer).

### Disable PII sampling

```yaml
# Disable sampling для PII-sensitive operators:
runtime-sampling.operator-blocklist: "user-pii-processor,payment-handler"
```

PII regulations (GDPR, CCPA) могут prohibit raw data sampling. Blocklist sensitive operators.

## Sample API: REST endpoint

Для programmatic access (CI/CD, automation):

```bash
# Request sample:
curl -X POST http://jobmanager:8081/jobs/<job-id>/vertices/<vertex-id>/sample \
  -H "Content-Type: application/json" \
  -d '{
    "subtask": 5,
    "direction": "output",
    "sampleSize": 100,
    "timeoutMs": 30000,
    "filter": "userId == \"u_whale_001\""
  }'

# Response:
{
  "requestId": "abc-123",
  "status": "PENDING"
}

# Poll for result:
curl http://jobmanager:8081/sampling-requests/abc-123
{
  "status": "COMPLETED",
  "records": [...],
  "metadata": {
    "sampledFrom": 1234567,
    "filterMatched": 100,
    "durationMs": 1200
  }
}
```

Polling pattern — sampling async, может занять секунды.

## Filter syntax

Simple expression language inspired by SQL:

```
userId = "u_001"
amount > 100
category IN ("books", "music")
timestamp > 1700000000000
userId IS NULL
```

Combinations:
```
amount > 100 AND category = "books"
userId = "u_001" OR userId = "u_002"
```

Note: Complex expressions evaluated per-record. На high-throughput filter cost может быть significant. Используйте simple filters where possible.

---

## Performance impact

Sampling design — **near-zero impact** на normal processing:

- Sample buffer fixed-size — no unbounded memory.
- Lightweight wrapper на mailbox — per-record overhead &lt;100ns.
- Sampling stops when buffer full или timeout — autodetach.
- Не blocking processing.

Empirically: &lt; 1% throughput drop during active sampling, no impact when inactive.

**Exception**: filter с heavy predicate может add 1-10% CPU (per-record evaluation). Quick samples — minor impact.

## Comparison с альтернативами

| Approach | Latency to insight | Production safety | Coverage |
|---|---|---|---|
| **Add log statement + redeploy** | 5-30 min (deploy cycle) | Risky (config change) | Limited (only post-deploy events) |
| **Persistent debug logging** | Continuous | Log flood, performance hit | Wide but noisy |
| **JFR profiling** | Setup time + analysis | Low overhead | CPU/memory only, не business data |
| **Custom debug branch в code** | Long deploy cycle | Code complexity | Wide but ugly |
| **FLIP-570 Runtime Sampling** | 1-2 seconds | Near-zero overhead | Real production events |

FLIP-570 — strictly better для ad-hoc debugging real production traffic.

## Limitations

### 1. No persistent storage
Samples — ad-hoc, не stored after viewing. Если хотите archive — download via UI или REST.

### 2. Не replaces metrics
Sampling — point-in-time view. Metrics — continuous trending. Both needed.

### 3. Sampling is sampling
100 records ≠ representative из million-throughput stream. Statistical inference careful.

### 4. Filter not Turing-complete
Только simple comparisons. No regex, no UDFs.

### 5. Не работает с serialized state
Sampling видит records, не state. Хотите inspect state — это отдельная фича (Queryable State, или State Processor API).

### 6. PII / compliance

Production teams должны think через compliance. PII в samples требует careful handling. Blocklist sensitive operators в config.

## Production patterns

### Pattern 1: Triage flow при incident

```
1. Alert: latency spike в operator X.
2. Open Web UI → operator X → busy 95%.
3. Click "Sample" на input subtask 0 — 100 records.
4. Identify pattern (skew, malformed data, etc.) — 1 minute.
5. Apply fix.
```

Standardize this flow в runbook. Junior on-call могут эффективно triage.

### Pattern 2: Pre-deploy verification

```
1. Deploy new version к staging.
2. Sample output operator X.
3. Verify format matches expectations.
4. Approve promote к production.
```

Sanity check без external test consumers.

### Pattern 3: Customer support escalation

```
Customer: "My account events are slow / missing."
Support: Sample with filter `userId = "<customer_id>"`.
Inspect: events visible? Throughput per user normal?
```

Bridge между customer report и engineering — пара минут.

### Pattern 4: Data quality continuous validation

```
CI job каждые 30 минут:
  curl ... /sample (no filter)
  Validate sampled records через schema/rules.
  Alert if violations > threshold.
```

Ongoing data quality monitoring без heavy stream processor.

## Production gotchas

### Gotcha 1: PII в samples

```
Sample output:
  {ssn: "123-45-6789", email: "user@example.com", ...}
```

PII leaked в UI / logs / shares. Compliance violation.

**Fix**: blocklist PII operators. Train team — sampling != free debugging для anything.

### Gotcha 2: Sampling production во время peak load

Filter с heavy predicate во время peak load может add CPU overhead → backpressure.

**Fix**: simple filters only во время peak. Save complex investigation для off-peak windows.

### Gotcha 3: Sample size > 1000 = большая UI download

10000 records × 5KB each = 50MB download → slow UI, possible timeout.

**Fix**: `max-records-per-request: 1000` — sane upper bound. Если нужно больше — REST API + scripts.

### Gotcha 4: Sampling доступен всем in K8s по умолчанию

Production Web UI без auth → любой может sample. Security risk.

**Fix**: enable access control + reverse proxy auth. Audit log sampling requests.

### Gotcha 5: Samples не captured во время GC pause

Если subtask в GC pause во время sample window, sample может быть incomplete. Re-sample.

### Gotcha 6: Filter expression syntax errors

```bash
filter: 'userId == "u_001"'   # Java-style ==
filter: 'userId = "u_001"'    # SQL-style =
```

Two syntaxes — verify which Flink version uses. Test simple filter first.

### Gotcha 7: Sampling после aggregation operator

Sample output после window aggregator — видите aggregated records, не raw events. Если нужно investigate raw — sample input window operator, не output.
