# EOS Survival Test — chaos lab с docker kill

Теоретическое понимание 2PC недостаточно для confidence в production EOS. Этот lab — практический chaos test: запустить pipeline с KafkaSink EOS, injected chaos в разные моменты transaction lifecycle, и **verify** что downstream Kafka topic содержит ровно те записи, что были произведены — без дубликатов, без потерь. Этот lab — finale модуля Exactly-Once и его pre-requisite для запуска EOS на production.

Chaos scenarios:
1. Kill TaskManager во время normal writes.
2. Kill TaskManager между prepareCommit и notifyCheckpointComplete.
3. Kill TaskManager после notifyCheckpointComplete но до commit.
4. Kill TaskManager во время barrier alignment.
5. Network partition между JobManager и TaskManager.
6. Slow `transaction.timeout.ms` — induce data loss intentionally.

<Callout type="tip">
EOS chaos test golden rule: kill TM между preCommit и commit — это самый коварный window. Verify zero duplicates + zero missing с consumer `isolation.level=read_committed`. Repeat scenario 100×, expect 100% pass для всех scenarios A-E.
</Callout>

## Test pipeline — deterministic producer + dedup consumer

### Producer — sequential integers

```java
public class DeterministicProducer {
    public static void main(String[] args) throws Exception {
        Properties props = new Properties();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, IntegerSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, LongSerializer.class);
        props.put(ProducerConfig.ACKS_CONFIG, "all");
        props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");

        KafkaProducer<Integer, Long> producer = new KafkaProducer<>(props);
        long n = 1_000_000;  // 1M messages
        for (long i = 0; i < n; i++) {
            producer.send(new ProducerRecord<>("input", (int)(i % 16), i));
            if (i % 1000 == 0) producer.flush();
        }
        producer.close();
        System.out.println("Produced " + n + " messages");
    }
}
```

Records: keys 0-15, values 0-999999 (sequential). Это deterministic — мы знаем что ровно эти числа должны выйти из pipeline.

### Flink pipeline — identity with state

```java
public class IdentityPipelineEos {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.enableCheckpointing(10_000);
        env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
        env.getCheckpointConfig().setCheckpointTimeout(60_000);
        env.setRestartStrategy(RestartStrategies.fixedDelayRestart(5, 5_000));

        KafkaSource<Long> source = KafkaSource.<Long>builder()
            .setBootstrapServers("kafka:9092")
            .setTopics("input")
            .setGroupId("eos-test")
            .setStartingOffsets(OffsetsInitializer.earliest())
            .setValueOnlyDeserializer(new LongDeserializationSchema())
            .build();

        DataStream<Long> stream = env.fromSource(source,
            WatermarkStrategy.noWatermarks(), "input-source");

        // KeyBy для stateful — usually есть какой-то stateful operator
        DataStream<Long> processed = stream
            .keyBy(v -> v % 16)
            .process(new IdentityWithState());

        KafkaSink<Long> sink = KafkaSink.<Long>builder()
            .setBootstrapServers("kafka:9092")
            .setRecordSerializer(KafkaRecordSerializationSchema.<Long>builder()
                .setTopic("output")
                .setValueSerializationSchema(new LongSerializationSchema())
                .build())
            .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
            .setTransactionalIdPrefix("eos-test-v1")
            .setProperty("transaction.timeout.ms", "3600000")
            .build();

        processed.sinkTo(sink);

        env.execute("EOS Survival Test");
    }

    static class IdentityWithState extends KeyedProcessFunction<Long, Long, Long> {
        ValueState<Long> lastSeen;

        @Override
        public void open(OpenContext ctx) {
            lastSeen = getRuntimeContext().getState(
                new ValueStateDescriptor<>("lastSeen", Long.class));
        }

        @Override
        public void processElement(Long value, Context ctx, Collector<Long> out) throws Exception {
            lastSeen.update(value);
            out.collect(value);
        }
    }
}
```

Identity pipeline (input == output), но содержит stateful operator чтобы ABS реально работал.

### Verifier — count and detect duplicates

```java
public class DuplicateVerifier {
    public static void main(String[] args) throws Exception {
        Properties props = new Properties();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "verifier");
        props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest");
        props.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG, "read_committed");  // КРИТИЧНО!
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, IntegerDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, LongDeserializer.class);

        KafkaConsumer<Integer, Long> consumer = new KafkaConsumer<>(props);
        consumer.subscribe(List.of("output"));

        Set<Long> seen = new HashSet<>();
        long duplicates = 0;
        long missing;
        long total = 0;
        long expectedTotal = 1_000_000;

        long deadline = System.currentTimeMillis() + 60_000;  // 1 min silence
        while (System.currentTimeMillis() < deadline) {
            ConsumerRecords<Integer, Long> records = consumer.poll(Duration.ofSeconds(1));
            if (!records.isEmpty()) {
                deadline = System.currentTimeMillis() + 60_000;
                for (ConsumerRecord<Integer, Long> r : records) {
                    total++;
                    if (!seen.add(r.value())) {
                        duplicates++;
                        System.err.println("DUPLICATE: " + r.value());
                    }
                }
            }
        }
        missing = expectedTotal - seen.size();
        System.out.printf("Total seen: %d, unique: %d, duplicates: %d, missing: %d%n",
            total, seen.size(), duplicates, missing);

        if (duplicates == 0 && missing == 0) {
            System.out.println("✓ EOS verified");
            System.exit(0);
        } else {
            System.out.println("✗ EOS violated");
            System.exit(1);
        }
    }
}
```

**Critical**: `isolation.level=read_committed`. Без этого consumer видит aborted transactions — невозможно отделить EOS violation от транзитного состояния.

<Callout type="warning">
Самая частая ошибка верификации EOS: consumer без `isolation.level=read_committed` видит aborted transactions как regular records → false positive «duplicates». Это объясняет 90% «EOS failures» в chaos tests. Всегда документировать downstream consumers: `read_committed` обязателен для EOS pipelines.
</Callout>

## Chaos test runner

```bash
#!/bin/bash
# chaos-test.sh — run pipeline + inject chaos + verify

set -e

echo "1. Start cluster..."
docker compose up -d
sleep 30

echo "2. Create topics..."
docker compose exec kafka kafka-topics.sh --bootstrap-server kafka:9092 \
    --create --topic input --partitions 16 --replication-factor 1
docker compose exec kafka kafka-topics.sh --bootstrap-server kafka:9092 \
    --create --topic output --partitions 16 --replication-factor 1

echo "3. Submit Flink job..."
docker compose exec jobmanager flink run -d /opt/flink/eos-pipeline.jar

echo "4. Start producer in background..."
java -jar producer.jar &
PRODUCER_PID=$!

echo "5. Wait 30s for steady state..."
sleep 30

echo "6. Inject chaos: kill TM during run..."
docker compose kill -s SIGKILL flink-eos-lab-taskmanager-1
sleep 60  # give recovery time

echo "7. Inject more chaos: kill again during commit cycle..."
sleep 5  # try to land kill mid-commit
docker compose kill -s SIGKILL flink-eos-lab-taskmanager-2
sleep 60

echo "8. Restart killed TMs..."
docker compose up -d taskmanager
sleep 30

echo "9. Wait for producer to finish..."
wait $PRODUCER_PID

echo "10. Wait for pipeline to drain..."
sleep 60

echo "11. Run verifier..."
java -jar verifier.jar
```

## Running individual scenarios

### Scenario A — normal kill

```bash
# Run producer producing 100k msgs
# Mid-flight kill one TM
docker compose kill -s SIGKILL flink-eos-lab-taskmanager-1
sleep 60
docker compose up -d taskmanager
# Wait для recovery, drain, verify
```

Expected: count == 100000, duplicates == 0.

### Scenario C — precise kill после preCommit

Это сложнее timing. Подход:
1. Set `execution.checkpointing.interval: 60s` — long interval.
2. В Flink Web UI watch checkpoint history.
3. Когда checkpoint(N) триггерится и acks accumulating → kill TM в ~3-5 секундный window между preCommit и notify.

Реализация через scripted timing:

```bash
# Watch JobManager logs для checkpoint events
docker compose logs -f jobmanager | grep -E "Triggering checkpoint|Completed checkpoint" &
LOGS_PID=$!

# Kill timing: после "Triggering checkpoint" + ~3s (preCommit done) перед "Completed checkpoint"
sleep 33  # adjust experimentally
docker compose kill -s SIGKILL flink-eos-lab-taskmanager-1
```

Realistic: 50% chance hit precise window. Repeat experiment N times.

### Scenario F — induce data loss

Intentional bad config:
```java
.setProperty("transaction.timeout.ms", "30000")  // 30s — too small
```

Steps:
1. Start pipeline с small timeout.
2. Run normal producer.
3. Kill TM — recovery from K8s pod restart takes 90s easily.
4. After 30s broker auto-aborts prepared transactions.
5. After 90s Flink recovers, calls commit on aborted txnIds → fails.
6. Verifier shows missing data.

Это **must-not-have** в production. Lab demonstrates **why** timeout setting matters.

## Production gotchas (lab learnings)

### 1. Consumer без read_committed → false positive duplicates

Если verifier пропустил `ISOLATION_LEVEL_CONFIG=read_committed`, он видит aborted transactions как regular records. Result: hundreds of «duplicates» которые на самом деле aborted txns invisible для real consumers.

**Fix**: всегда set `isolation.level=read_committed` для верификации EOS.

### 2. Kill только TaskManager — not JobManager

Killing JM может вызвать другой failure mode — entire cluster down, recovery from CompletedCheckpointStore (ZooKeeper/K8s ConfigMap). Это more disruptive scenario, testing HA не EOS.

**Fix**: для EOS test — kill только TM. Для HA test — kill JM.

### 3. Kafka cluster size — single broker для lab

Single broker простой setup, но `transaction.state.log.replication.factor=1` — single point of failure. Не representative для production multi-broker.

**Fix**: для production-realistic test — 3 brokers с replication 3, ISR 2, и kill один broker mid-flight.

### 4. Producer overrun checkpoint interval

Если producer rate > pipeline throughput, backlog grows. Каждый checkpoint больше предыдущего, time grows. Eventually checkpoint timeout.

**Fix**: для lab — bounded producer (1M records). Для real-world test — throttle producer match downstream throughput.

### 5. Verifier early exit

Если verifier `poll` timeout пока pipeline still in recovery, может exit prematurely с false «missing». 

**Fix**: long enough silence timeout (60s+), or wait для kafka log consumer-offsets advance to producer offset.

### 6. State backend — RocksDB vs HashMap

RocksDB checkpoints (incremental) recover быстрее на large state. HashMap (heap) faster на small state. Lab с large state требует RocksDB; small state OK с HashMap.
