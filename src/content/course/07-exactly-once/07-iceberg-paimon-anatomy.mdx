# Iceberg и Paimon sinks — анатомия 2PC через manifests

Iceberg и Paimon — два главных streaming-friendly lakehouse формата 2026. Оба integrate в Flink через SinkV2 API и реализуют exactly-once через **manifest-based commits**: data files написаны и явно зарегистрированы атомарно через transactional metadata layer. Этот урок препарирует sink anatomy для обоих и показывает, почему architectural decomposition SinkV2 (Writer + PreCommit + Committer + PostCommit) идеально fit на эту задачу.

Apache Iceberg (proven в Netflix, Apple) и Apache Paimon (рождён в Flink community, optimized для streaming) имеют разные internals, но **same SinkV2 wiring**.

<Callout type="tip">
Iceberg/Paimon sinks ОБЯЗАТЕЛЬНО используют `TwoPhaseCommittingSink` + `SupportsPostCommitTopology`. Post-commit для manifest publish / catalog update. Pre-commit для small file compaction — без неё manifest entries growth = query planning latency растёт линейно.
</Callout>

- Writer parallelism — proportional throughput
- Committer parallelism — typically 1 (atomic commits)
- Pre/post topology — independent

## Iceberg sink — internals

Iceberg model: **table = list of snapshots**. Each snapshot points to **manifest list**, which points to **manifest files**, which point to **data files**.

```
catalog (Glue, Hive, Nessie, JDBC)
    ↓ pointer to current metadata.json
metadata/v3.metadata.json
    ↓ list of snapshots
[snapshot-1, snapshot-2, snapshot-3]
    ↓ each → manifest list
metadata/snap-{snapshotId}.avro
    ↓ list of manifest files
[manifest-a.avro, manifest-b.avro]
    ↓ each → list of data files + stats
data/00000-0-{uuid}.parquet
data/00001-0-{uuid}.parquet
```

Commit = atomic CAS-update catalog pointer to new `metadata.json`. Это single point of atomicity.

### Iceberg FlinkSink (org.apache.iceberg.flink)

```java
import org.apache.iceberg.flink.sink.FlinkSink;

FlinkSink.forRowData(dataStream)
    .tableLoader(TableLoader.fromCatalog(catalogLoader, TableIdentifier.of("db", "events")))
    .writeParallelism(16)
    .equalityFieldColumns(Arrays.asList("user_id"))  // upsert mode
    .overwrite(false)                                 // append mode
    .distributionMode(DistributionMode.HASH)
    .append();
```

Internally:
1. **IcebergStreamWriter** (SinkWriter) — writes Parquet per partition spec, emits `WriteResult` (data files list).
2. **IcebergFilesCommitter** (Committer) — receives WriteResults, accumulates buffer, on `notifyCheckpointComplete` calls Iceberg API `table.newAppend().appendFile(...).commit()`.
3. Iceberg API делает CAS update metadata pointer:
   ```
   table.refresh();  // get current metadata
   AppendFiles append = table.newAppend();
   for (DataFile f : files) append.appendFile(f);
   append.commit();  // CAS update catalog
   ```
4. CAS conflict → retry (другой writer обновил catalog между read и write).

### Equality deletes — для upsert

Iceberg v2 supports **equality deletes**: delete files with `(user_id, event_time)` patterns. Flink sink с `equalityFieldColumns` пишет equality delete records при upsert mode, plus data files. На read merge-on-read joins delete + data.

```java
.equalityFieldColumns(Arrays.asList("user_id"))
```

Каждый record `delete(user_id=X)` + `insert(user_id=X, ...)`. Старая запись помечена deleted, новая appended. Compaction позже physically merge.

## Paimon sink — internals

Paimon (formerly Flink Table Store) — born в Flink community, designed для streaming-first. Внутри LSM-tree поверх Parquet/ORC.

### Architecture differences vs Iceberg

| Aspect | Iceberg | Paimon |
|---|---|---|
| Storage | flat list of immutable files | LSM-tree (levels с compaction) |
| Update | equality deletes (merge-on-read) | UPSERT через primary key, native merge |
| Snapshot | metadata.json + manifest list | snapshot file + manifest |
| Read | full scan + merge deletes | sorted merge через LSM |
| Compaction | external action | continuous, integrated |
| Streaming read | через snapshot deltas | through changelog producer |

### Paimon FlinkSink

```java
import org.apache.paimon.flink.sink.FlinkSinkBuilder;

CatalogContext context = CatalogContext.create(new Path("s3://lakehouse/paimon"));
Catalog catalog = CatalogFactory.createCatalog(context);
Table table = catalog.getTable(Identifier.create("db", "events"));

FlinkSinkBuilder.<RowData>forTable(table)
    .withInput(dataStream)
    .withParallelism(16)
    .build();
```

Internally:
1. **StoreWriteOperator** — writes records в LSM MemTable. На barrier flushes к L0 files.
2. **StoreCommitOperator** — collects committables, на `notifyCheckpointComplete` bumps snapshot ID, writes snapshot manifest pointing to new files.
3. Paimon атомарность через **filesystem rename** snapshot file:
   ```
   write snapshot-{N+1} to temp path
   rename to snapshot/{N+1}
   update LATEST file (atomic file write)
   ```

### Compaction — integrated

Paimon writers могут trigger inline compaction:

```yaml
table.options:
  compaction.min.file-num: 5
  compaction.max.file-num: 50
  compaction.early-max.file-num: 12
```

Если L0 > threshold — writer thread compacts L0→L1 sync или async. Это **отличается** от Iceberg, где compaction = separate procedure.

## SinkV2 wiring — Iceberg example

Полная реализация Iceberg FlinkSink через SinkV2 interfaces:

```java
public class IcebergSink<T> implements
    TwoPhaseCommittingSink<T, IcebergCommittable>,
    SupportsPreCommitTopology<IcebergWriteResult, IcebergCommittable>,
    SupportsPostCommitTopology<IcebergCommittable> {

    @Override
    public PrecommittingSinkWriter<T, IcebergWriteResult> createWriter(WriterInitContext ctx) {
        return new IcebergStreamWriter<>(tableSupplier, taskWriterFactory);
    }

    @Override
    public Committer<IcebergCommittable> createCommitter(CommitterInitContext ctx) {
        return new IcebergFilesCommitter(tableSupplier, catalogProperties);
    }

    @Override
    public DataStream<IcebergCommittable> addPreCommitTopology(
            DataStream<IcebergWriteResult> writeResults) {
        return writeResults
            .keyBy(IcebergWriteResult::getPartitionPath)
            .transform("compaction",
                IcebergCommittableTypeInfo.INSTANCE,
                new SmallFileCompactionOperator(targetFileSize))
            .setParallelism(8);
    }

    @Override
    public void addPostCommitTopology(DataStream<IcebergCommittable> committed) {
        committed
            .global()  // single subtask чтобы catalog не race
            .transform("manifest-publish",
                Types.VOID,
                new ManifestPublishOperator(hiveCatalogConfig))
            .setParallelism(1);
    }
}
```

Каждый stage — independent operator с own state, own parallelism.

## Catalog choices — где живёт metadata

### Iceberg catalogs

| Catalog | Storage | Atomicity | Production-ready |
|---|---|---|---|
| Hive | Hive metastore Postgres/MySQL | DB transaction | ✅ legacy |
| Glue (AWS) | AWS Glue Data Catalog | AWS internal lock | ✅ AWS |
| Nessie | Git-like branching catalog | DB transaction | ✅ для multi-table txns |
| JDBC | any RDBMS | DB transaction | ✅ K8s simple |
| REST | HTTP service | server-managed | ✅ multi-tenant |

Best choice: Nessie если нужны branches/tags (data version control), Glue если на AWS, JDBC для simple setup.

### Paimon catalogs

| Catalog | Storage |
|---|---|
| Filesystem (HDFS/S3) | only filesystem — atomic rename |
| Hive | Hive metastore reference |
| JDBC | any RDBMS |

Paimon филесистем catalog работает в standalone setups (single source-of-truth = filesystem).

## Iceberg vs Paimon — when to use what

**Iceberg**:
- Multi-engine read (Spark, Trino, Snowflake, Flink) — широчайшая совместимость
- Slowly changing dim tables — append-heavy с occasional updates
- Historical data lake — time travel, schema evolution
- AWS S3-first deployments — Glue + Iceberg = mature combo

**Paimon**:
- Streaming-first workloads — created from Flink team
- High update rate (CDC sink) — LSM-tree handles updates natively
- Need changelog stream — Paimon emits changelog для downstream Flink consumers
- Latency-sensitive (snapshot every few minutes) — LSM compaction integrated

В практике многие team используют ОБА: Paimon для streaming pipelines, Iceberg для exploratory analytics. Или Paimon → Iceberg conversion для long-term storage.

## Production gotchas

### 1. Committer parallelism > 1 → catalog race

Если установить committer parallelism к 8, multiple subtasks parallelly call `table.newAppend().commit()` → catalog CAS conflicts → most retry, slow throughput.

**Fix**: committer parallelism = 1. Iceberg/Paimon defaults to это.

<Callout type="warning">
Committer parallelism MUST = 1 для Iceberg/Paimon. Multiple subtasks parallelly call `table.newAppend().commit()` → catalog CAS conflicts → 99% retry, throughput cratters. Если задавали parallelism > 1 manually — откатить.
</Callout>

### 2. Без pre-commit compaction → millions of small files

64MB Parquet × 16 subtasks × 60 checkpoints/hour × 24h = 22GB но 23,000 файлов. Query latency degrades 10-100x.

**Fix**: добавить pre-commit compaction. Target size 128-256MB per file.

### 3. Glue catalog rate limits

AWS Glue имеет rate limits (50 calls/sec per account). Committer на каждый checkpoint делает несколько Glue calls. С 12 jobs, каждый checkpoint каждые 60s — close to limit. Burst может throttle.

**Fix**: либо реже checkpoints (5min interval), либо REST catalog с Apache Polaris.

### 4. Paimon LSM compaction lag

Paimon writers могут sync-compact L0→L1 — это блокирует write thread. Если compaction expensive (CPU/IO bound), throughput drops.

**Fix**: configure `compaction.async=true` (write thread continues, compaction в background). Может lead к temporary L0 buildup.

### 5. Iceberg equality deletes на full table scan

Equality deletes мердж-on-read — каждый scan читает delete files + data files, applies deletes. На high update rate (millions deletes), scan slow.

**Fix**: regular `RewriteFiles` action для merge deletes в data files (physical delete). Schedule daily/hourly.

### 6. Metadata.json explosion

Каждый snapshot adds entry в metadata.json. После 100k snapshots — metadata file MB-size, slow to read.

**Fix**: `table.expireSnapshots()` regular cleanup. Iceberg `metadata.retain-last` config = 100. Снимки старше 7 days physically deleted.

### 7. PostCommit operations не EOS

Если post-commit Hive metastore update fails и restart → recommit для already-committed snapshot. Hive add partition должен быть idempotent (`IF NOT EXISTS`).

**Fix**: idempotent post-commit operations. Wrap external calls в try-catch для duplicate handling.

## Cheat sheet

```
Iceberg sink anatomy (SinkV2):
    Writer  : Parquet files per partition spec
    PreCommit: compact small files (10×64MB → 1×256MB)
    Committer: table.newAppend().commit() → atomic snapshot
    PostCommit: Hive update, manifest rewrite

Paimon sink:
    Writer  : LSM MemTable → L0 SST на barrier
    PreCommit: optional compaction
    Committer: snapshot file write + rename
    PostCommit: notifications

Catalog atomicity:
    Iceberg: CAS на metadata.json pointer
    Paimon : atomic filesystem rename snapshot file

Committer parallelism = 1 (avoid catalog race)
```
