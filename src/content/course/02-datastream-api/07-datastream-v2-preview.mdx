# DataStream API V2 (FLIP-409): preview

В Flink 2.0 (март 2025) представлен **DataStream API V2 (FLIP-409)** — переосмысление DataStream API после 9 лет накопленного опыта. Это experimental preview, **не для production до Flink 2.3-2.4**, но важно понимать direction. Через 1-2 версии V2 станет default, V1 окажется в legacy mode (как DataSet API в своё время).

В этом уроке: motivation для V2, ключевые abstractions (DataStream, ProcessFunction, Partitioning + extensions), сравнение V1 vs V2, и production-планирование migration.

---

## Зачем DataStream V2

DataStream API родился в 2014 (Flink 0.9). Девять лет эволюции наслаивало решения друг на друга, что привело к ряду проблем:

1. **Concept overload** — KeyedStream, DataStream, WindowedStream, ConnectedStreams, BroadcastStream, JoinedStream — six different abstractions for related concepts. Cognitive load выше необходимого.
2. **Tight coupling watermarks** — watermark machinery встроен глубоко. Бounded streams ведут себя странно, потому что API designed для streaming-first.
3. **Mixed batch / streaming** — после unified `Boundedness`, watermark mechanics стали awkward для bounded jobs.
4. **Implicit вещи** — typing, partitioning, state — много implicit poведения, hard to introspect.
5. **Constraints on extensions** — добавить новый kind of operation требует API surface change.

V2 цель: **chist, composable, explicit API**.

### DataStream

Один main type вместо иерархии. Operations возвращают `DataStream<T>`:

```java
// V2 syntax (preview, может измениться)
DataStream<Tx> txs = env.fromSource(kafkaSource, ...);

DataStream<EnrichedTx> enriched = txs
    .process(new EnrichmentProcess());  // одно — process() для всех типов processing

DataStream<Alert> alerts = enriched
    .partitionBy(EnrichedTx::getUserId)  // partitioning отдельно от processing
    .process(new FraudDetectionProcess());
```

### ProcessFunction (V2)

Единый интерфейс — без 12 разных подклассов:

```java
// V2 ProcessFunction (preview)
public class FraudDetection extends ProcessFunction<EnrichedTx, Alert> {

    private StateDescriptor<ValueState<UserHistory>> historyState =
        ValueStateDescriptor.of("history", UserHistory.class);

    @Override
    public void open(OpenContext ctx) {
        ctx.requireKeyed();  // declare что нужно keyed state
    }

    @Override
    public void processRecord(
            EnrichedTx record,
            ProcessContext ctx,
            Collector<Alert> out) {
        // Доступ к state, timers, side outputs
        UserHistory history = ctx.state(historyState).value();
        // ... logic
    }

    @Override
    public void onTimer(long timestamp, OnTimerContext ctx, Collector<Alert> out) {
        // unchanged
    }
}
```

### Partitioning explicit

В V1: `keyBy()` создавал `KeyedStream` — implicit type. В V2 partitioning — explicit parameter:

```java
// V1 (current):
stream.keyBy(Tx::getUserId).process(new MyProcess());  // implicit KeyedStream

// V2 (preview):
stream
    .partitionBy(Tx::getUserId)  // partitioning explicit
    .process(new MyProcess());

// Или для не-keyed processing:
stream
    .partitionBy(Partitioning.broadcast())
    .process(new MyBroadcastProcess());
```

Это даёт benefits:

- **Composable** — partitioning сам по себе abstraction.
- **Multiple partitioning strategies** — broadcast, custom hash, range — все uniform.
- **Type clear** — `DataStream<T>` остаётся одним типом независимо от partitioning.

## V1 vs V2 — comparison

| Aspect | V1 (current) | V2 (preview) |
|---|---|---|
| Main abstractions | DataStream, KeyedStream, ConnectedStreams, WindowedStream, BroadcastStream | DataStream + ProcessFunction + Partitioning |
| ProcessFunction variants | 12+ subclasses | 1 unified |
| Partitioning | Implicit через keyBy() | Explicit parameter |
| Watermark machinery | Coupled с DataStream | Decoupled, optional |
| Bounded / unbounded | Same API but semantics differ | Unified properly |
| Window API | Built-in | Extension |
| Joins API | Built-in | Extension |
| Adding new operations | Requires core changes | Through extensions |
| Type safety | Java generics + reflection | Improved |

## Что НЕ меняется

Важно понять, что V2 — это эволюция API, не разрушение фундамента:

- **Runtime** — JobManager / TaskManager / slots / scheduling неизменно.
- **State backends** — RocksDB / ForSt одинаковые.
- **Checkpointing / Recovery** — ABS алгоритм, savepoints, 2PC — те же.
- **Connectors** — FLIP-27 sources, FLIP-191 sinks работают с V1 и V2.
- **Table API / SQL** — отдельный API, не affected.
- **Deployment** — Application/Session mode, K8s Operator — те же.

Это значит: знание Flink fundamentals (что мы изучаем в этом курсе) полностью transferable на V2. Меняется только syntax layer.

## Migration strategy

Когда V2 будет stable (предположительно 2.4-2.5), migration:

1. **Interoperability layer** — V1 и V2 будут coexist в одном job. Можно постепенно migrate operator-by-operator.
2. **Automatic conversion** — для simple jobs (без custom ProcessFunction) — программный rewrite tool.
3. **Savepoint compatibility** — state from V1 jobs должен restore в V2 (с UIDs обязательно). Это критическое требование.
4. **API parity** — V2 будет полностью покрывать V1 use cases прежде чем V1 deprecated.

Production strategy:

```
Phase 1 (now): use V1, follow current course
Phase 2 (~2.3 release): experiment с V2 в development environments
Phase 3 (~2.5 release): new jobs на V2
Phase 4 (~3.0): migrate existing V1 jobs к V2
```

## DataStream V2 sample (preview API)

Внимание: API в alpha, syntax может measure-измениться. Это **conceptual preview**, не final syntax.

```java
// Hypothetical V2 syntax — для educational purposes
public class FraudDetectionV2 {

    public static void main(String[] args) throws Exception {
        ExecutionEnvironment env = ExecutionEnvironment.getEnvironment();
        env.enableCheckpointing(60_000);

        DataStream<Transaction> txs = env.fromSource(
            KafkaSource.builder().build(),
            "transactions");

        DataStream<EnrichedTx> enriched = txs
            .process(new EnrichmentFunction());

        DataStream<Alert> alerts = enriched
            .partitionBy(EnrichedTx::getUserId)
            .process(new FraudDetectionFunction());

        alerts.sinkTo(KafkaSink.builder().build());

        env.execute("Fraud Detection V2");
    }
}

class FraudDetectionFunction extends ProcessFunction<EnrichedTx, Alert> {

    private static final StateDescriptor<ValueState<UserState>> STATE =
        ValueStateDescriptor.of("user-state", UserState.class);

    @Override
    public void configure(ProcessFunctionContext ctx) {
        ctx.requireKeyed();
        ctx.declareState(STATE);
    }

    @Override
    public void processRecord(EnrichedTx record, ProcessContext ctx,
                              Collector<Alert> out) {
        UserState state = ctx.state(STATE).value();
        // ... fraud logic
    }
}
```

## Production gotchas

- **Не используйте V2 в production до 2.3-2.4** — experimental status означает breaking changes между minor versions. Production миграция на experimental API — recipe для боли.
- **Изучайте V1 fundamentals** — это transferable knowledge. KeyedProcessFunction concepts, state, timers, side outputs — всё это будет в V2.
- **State migration** — V1 → V2 будут поддерживать savepoints, но с UIDs обязательно. Без UIDs на V1 — потеря state при V2 migration. Set UIDs **NOW** на V1 jobs.
- **Track FLIP-409 progress** — Flink mailing list, release notes. По мере stabilization обновляйте опытные jobs.
- **Mixed V1+V2 jobs** — будут поддерживаться, но потенциально complex для debugging. Лучше migrate целым.

<Callout type="warning">
**Hype warning**: некоторые conference talks 2025 могут представлять V2 как "ready". Reality: 2.0 release — preview. Production-ready timeline 2.3-2.4 (2026). Не торопитесь.
</Callout>

## Comparison: similar API evolutions

Flink не первый, кто делает major API rewrite:

| Project | V1 | V2 | Migration time |
|---|---|---|---|
| **Spark DataFrame** | RDD API | DataFrame API | 2-3 года coexistence |
| **Spark Streaming** | DStream API | Structured Streaming | 2-3 года |
| **Flink DataSet** | DataSet API | Unified DataStream (Bounded) | 5+ years deprecation, removed 2.0 |
| **Flink DataStream** | V1 | V2 (FLIP-409) | Estimate 2-3 года |

Pattern: stable production использует old API ещё долго после introduction нового. Это нормально.
