# Checkpointing и Savepoints — обзор модуля (KILLER)

Checkpointing — механизм fault tolerance Flink. Без checkpoints Flink был бы at-most-once stream processor. С ABS алгоритмом (Asynchronous Barrier Snapshotting) и transactional sinks Flink — exactly-once на TBs of state с миллисекундной recovery. Этот модуль — самый алгоритмически глубокий в курсе.

## ABS Algorithm — главный алгоритм

1. CheckpointCoordinator инжектит **CheckpointBarrier(N)** во все sources
2. Barrier течёт по DAG как обычная запись (flush немедленно)
3. Source: snapshot state → forward barrier
4. Operator с N input channels:
   - Получил barrier на A → блокирует A, буферизирует
   - Продолжает читать B, C, ... пока не получит barrier на всех
   - **Alignment** complete → snapshot → forward barrier → unblock
5. Sink: barrier на всех → snapshot → ack JM
6. Все acks → checkpoint complete → callback `notifyCheckpointComplete`

### Sync vs Async phases

- **Sync phase** (barrier пришёл) — immutable snapshot создаётся (COW / RocksDB native snapshot — hardlinks на SST файлы)
- **Async phase** — фактическая загрузка в DFS (processing продолжается!)

## Aligned vs Unaligned

| Aspect | Aligned | Unaligned (FLIP-76, 1.11+) |
|---|---|---|
| Под backpressure | Долгое alignment | Barrier обгоняет in-flight |
| Snapshot size | Без in-flight | + ChannelState (in-flight buffers) |
| Recovery cost | Дешёвая | Дороже (replay channel state) |
| Best for | Низкая backpressure | Высокая backpressure |

**Hybrid (1.14+)**: `execution.checkpointing.unaligned: true` + `aligned-checkpoint-timeout: 30s` — start aligned, escalate to unaligned если alignment > timeout.

## Generic Log-Based Incremental Checkpoints (FLIP-158)

**Проблема**: даже async + incremental, время checkpoint зависит от RocksDB flush cycle. Нестабильное.

**Решение** — **Changelog State Backend** + **DSTL**:
- Каждый state mutation дубль-пишется: в RocksDB + в DSTL (append-only log на DFS)
- DSTL flush на checkpoint → ms (маленький append)
- **Materialization** — periodic (default 10min): RocksDB snapshot → DFS → DSTL truncate

```yaml
state.backend.changelog.enabled: true
state.backend.changelog.storage: filesystem
dstl.dfs.base-path: s3://my-bucket/dstl
state.backend.changelog.periodic-materialize.interval: 10min
```

**Что даёт**: checkpoint interval seconds на TB-scale state, stable e2e latency для EOS sinks (Kafka transactions короче).

## Savepoints vs Checkpoints

| | Checkpoint | Savepoint |
|---|---|---|
| Trigger | Auto, periodic | Manual |
| Format | Native | Canonical OR Native (1.15+) |
| Purpose | Fault tolerance | Planned upgrade |
| Lifecycle | Owned by Flink | Owned by user |
| Backend switch | ❌ | ✅ (canonical) |

## Killer takeaway

Включайте **`state.backend.changelog.enabled: true`** для любой job с state > 10GB. Это решает production-essential проблему: stable checkpoint latency. Особенно критично для EOS sinks — короткие Kafka transactions выигрывают.
